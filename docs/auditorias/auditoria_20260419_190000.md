## VIOLACIONES CRÍTICAS (🔴)

---

### 1. **[CRÍTICA]**: Fuga de firma digital en logs

- **[CATEGORÍA]**: Seguridad (A02 - Cryptographic Failures)
- **[DESCRIPCIÓN]**: Se loguea parte de la firma digital (20 caracteres) en `GenerarContratoAdhesionUseCase`. Las firmas digitales son secretos criptográficos que NO deben aparecer en ningún log.
- **[ARCHIVO]**: `backend/src/main/java/com/tufondo/documentospdf/application/usecase/GenerarContratoAdhesionUseCase.java:124-125`
- **[IMPACTO]**: Un atacante con acceso a logs podría obtener fragmentos de firmas digitales RSA. Aunque sean 20 caracteres, esto reduces la entropía del secreto y podría facilitar ataques de fuerza bruta o análisis criptográfico.
- **[RECOMENDACIÓN]**:
```java
// ANTES (VULNERABLE):
log.info("Contrato de adhesión generado exitosamente: documentoId={}, firmaDigital={}",
        documento.getId(), firmaDigital.substring(0, 20) + "...");

// DESPUÉS (CORRECTO):
log.info("Contrato de adhesión generado exitosamente: documentoId={}, tipoFirma=RSA-SHA256, tieneFirma={}",
        documento.getId(), firmaDigital != null);
```

### 2. **[CRÍTICA]**: Flujo CS-004 VIOLADO - Hash calculado ANTES de escaneo ClamAV

- **[CATEGORÍA]**: Seguridad (A08 - Software and Data Integrity Failures)
- **[DESCRIPCIÓN]**: En los UseCases de ESTADO_CUENTA, CONSTANCIA_AFILIACION, TABLA_AMORTIZACION y CARTA_BENEFICIARIOS, el orden es:
  1. Generar PDF
  2. **Calcular hash SHA-256**
  3. Escanear con ClamAV

El orden correcto según CS-004 es: Generar PDF → **Escanear ClamAV PRIMERO** → Calcular hash.

- **[ARCHIVO]**: `backend/src/main/java/com/tufondo/documentospdf/application/usecase/GenerarEstadoCuentaUseCase.java:91-95`
- **[IMPACTO]**: Si el PDF generado es malicioso, el hash de contenido malicioso se registra como válido en la base de datos. El malware queda "certificado" con un hash SHA-256 limpio en el sistema.
- **[RECOMENDACIÓN]**:
```java
// ANTES (VULNERABLE):
// 5. Generar PDF
byte[] pdfBytes = pdfGeneratorPort.generarPdf(TipoDocumento.ESTADO_CUENTA, datosPdf);
// 6. Calcular hash SHA-256  <-- MAL
String hash = calcularHashSha256(pdfBytes);
// 7. Escanear con ClamAV   <-- MUY MAL (después de hash)
MalwareScannerPort.ScanResult scanResult = malwareScannerPort.scan(pdfBytes);

// DESPUÉS (CORRECTO):
// 5. Generar PDF
byte[] pdfBytes = pdfGeneratorPort.generarPdf(TipoDocumento.ESTADO_CUENTA, datosPdf);
// 6. Escanear con ClamAV PRIMERO  <-- CORRECTO
MalwareScannerPort.ScanResult scanResult = malwareScannerPort.scan(pdfBytes);
if (scanResult.isMalicious()) {
    log.error("PDF detectado como malicioso: {}", scanResult.threatName());
    throw new EscaneoMalwareException("PDF malicioso: " + scanResult.threatName());
}
// 7. Calcular hash SHA-256 (del contenido limpio)
String hash = calcularHashSha256(pdfBytes);
```

**Archivos afectados:**
- `GenerarEstadoCuentaUseCase.java:91-95`
- `GenerarConstanciaAfiliacionUseCase.java:80-84`
- `GenerarTablaAmortizacionUseCase.java:90-94`
- `GenerarCartaBeneficiariosUseCase.java:103-107`

### 3. **[CRÍTICA]**: Doble firma digital con algoritmos diferentes + hash inconsistente

- **[CATEGORÍA]**: Seguridad (A08 - Software and Data Integrity Failures)
- **[DESCRIPCIÓN]**: Los UseCases CONTRATO y PAGARE tienen dos mecanismos de firma冲突:
  1. `OpenPdfGeneratorService.firmarPdf()` usa keystore PKCS12 (línea 330-361)
  2. `GenerarContratoAdhesionUseCase.firmarPdf()` usa clave Base64 injectada (línea 164-189)

Además, los UseCases calculan hash SHA-256 del PDF YA FIRMADO por el servicio, pero la firma se hace ANTES del escaneo ClamAV en OpenPdfGeneratorService.

- **[ARCHIVO]**: `OpenPdfGeneratorService.java:183-193` y `GenerarContratoAdhesionUseCase.java:79-99`
- **[IMPACTO]**: Se aplican dos firmas digitales diferentes al mismo documento. El hash almacenado es del PDF con la primera firma, pero la segunda firma se aplica sobre un documento ya firmado. Esto rompe la verificabilidad de la firma y la trazabilidad del documento.
- **[RECOMENDACIÓN]**:
  1. Unificar el mecanismo de firma: O usar keystore en OpenPdfGeneratorService O usar clave Base64 en UseCases, nunca ambos.
  2. Mover la firma digital AL USE CASE (después del escaneo ClamAV), no dentro del generador de PDF.
  3. El hash debe calcularse del PDF SIN firmar, y la firma se aplica después del escaneo.

### 4. **[CRÍTICA]**: Log de error de firma digital expone configuración interna

- **[CATEGORÍA]**: Seguridad (A05 - Security Misconfiguration)
- **[DESCRIPCIÓN]**: Los mensajes de error en `firmarPdf()` exponen detalles de configuración interna:
  - "FIRMA DIGITAL REQUERIDA para CONTRATO pero keystore no configurado"
  - "DOC_005: Keystore de firma digital no configurado"

- **[ARCHIVO]**: `GenerarContratoAdhesionUseCase.java:167-169` y `GenerarPagareUseCase.java:165-167`
- **[IMPACTO]**: Un atacante que reciba estos errores知道她 el sistema no tiene firma digital configurada, información útil para planificar ataques.
- **[RECOMENDACIÓN]**:
```java
// ANTES (VULNERABLE):
log.error("FIRMA DIGITAL REQUERIDA para CONTRATO pero keystore no configurado");

// DESPUÉS (CORRECTO):
log.error("DOC_005: No se pudo generar el documento. Contacte al administrador.");
// NO revelar detalles de configuración en logs ni en respuesta al cliente
```

## VIOLACIONES ALTAS (🟠)

---

### 5. **[ALTA]**: Arquitectura Clean Architecture violada - Controller en infraestructura

- **[CATEGORÍA]**: Arquitectura
- **[DESCRIPCIÓN]**: El `DocumentoController` está en `infrastructure/presentation/controller/` cuando debería estar en `presentation/controller/` según la estructura de capas limpias. La capa de presentación no debería depender de infraestructura.
- **[ARCHIVO]**: `backend/src/main/java/com/tufondo/documentospdf/infrastructure/presentation/controller/DocumentoController.java`
- **[IMPACTO]**: Acoplamiento incorrecto. Si se cambia de Spring a otro framework, debería poder reutilizar la capa de presentación. Actualmente está atada a Spring MVC por estar en infraestructura.
- **[RECOMENDACIÓN]**: Mover el Controller a `presentation/controller/DocumentoController.java`

---

### 6. **[ALTA]**: Ubicación incorrecta de enums de dominio

- **[CATEGORÍA]**: Arquitectura
- **[DESCRIPCIÓN]**: Los enums están en `domain/model/enums/` cuando deberían estar en `domain/enums/` directamente.
- **[ARCHIVO]**:
  - `backend/src/main/java/com/tufondo/documentospdf/domain/model/enums/TipoDocumento.java`
  - `backend/src/main/java/com/tufondo/documentospdf/domain/model/enums/EstadoDocumento.java`
  - `backend/src/main/java/com/tufondo/documentospdf/domain/model/enums/ClasificacionDocumento.java`
- **[IMPACTO]**: Los enums son conceitos de dominio puro. Mezclarlos dentro de `domain/model/` sugiere que son parte del modelo de dominio cuando en realidad son valores independientes.
- **[RECOMENDACIÓN]**: Crear paquete `domain/enums/` y mover los tres archivos allí.

---

### 7. **[ALTA]**: Código duplicado - calcularHashSha256()

- **[CATEGORÍA]**: Mantenibilidad
- **[DESCRIPCIÓN]**: El método `calcularHashSha256()` está duplicado en 6 UseCases con implementación idéntica.
- **[ARCHIVO]**: Todos los UseCases tienen este método copy-paste:
  - `GenerarContratoAdhesionUseCase.java:144-158`
  - `GenerarPagareUseCase.java:146-160`
  - `GenerarEstadoCuentaUseCase.java:144-158`
  - `GenerarConstanciaAfiliacionUseCase.java:131-145`
  - `GenerarTablaAmortizacionUseCase.java:141-155`
  - `GenerarCartaBeneficiariosUseCase.java:154-168`
- **[IMPACTO]**: Violación DRY. Si se necesita cambiar el algoritmo de hash, hay que modificar 6 lugares. Riesgo de inconsistencias.
- **[RECOMENDACIÓN]**: Crear un servicio de dominio `HashingService` en `domain/service/` o un Util en `application/util/`.

---

### 8. **[ALTA]**: Posible estado inconsistente en documento si falla después de subir a MinIO

- **[CATEGORÍA]**: Seguridad
- **[DESCRIPCIÓN]**: En todos los UseCases, el flujo es:
  1. Generar PDF
  2. Escanear/Hash/Firmar
  3. Subir a MinIO
  4. Guardar documento en BD

Si el paso 3 falla, no hay problema (no se guardó nada). Pero si el paso 3 succeeds y paso 4 falla, el archivo está en MinIO pero no en BD, creando un orphaned object.

- **[ARCHIVO]**: Todos los UseCases (ej. `GenerarEstadoCuentaUseCase.java:101-121`)
- **[IMPACTO]**: Archivos huérfanos en MinIO que consumen almacenamiento pero no son rastreables por la BD.
- **[RECOMENDACIÓN]**: Usar el patrón Transactional Outbox o implementar limpieza periódica de archivos sin referencia en BD.

---

### 9. **[ALTA]**: No se valida que los datos sensibles estén enmascarados en PDFs

- **[CATEGORÍA]**: Compliance LOPDP/SUDEBAN
- **[DESCRIPCIÓN]**: No hay validación de que datos como cédulas, números de cuenta, etc. estén enmascarados antes de insertarlos en el PDF. El método `maskSocioId()` en OpenPdfGeneratorService solo enmascara UUIDs, no datos financieros.
- **[ARCHIVO]**: `OpenPdfGeneratorService.java:394-399`
- **[IMPACTO]**: Posible violación de LOPDP si números de cédula o datos financieros aparecen completos en PDFs que podrían ser exposés.
- **[RECOMENDACIÓN]**: Implementar servicios de enmascaramiento de datos y usarlos en los métodos `generarXxx()` antes de pasar datos al PDF.

### 10. **[MEDIA]**: El método `firmarPdf()` en UseCases no limpia la clave privada de memoria

- **[CATEGORÍA]**: Rendimiento/Seguridad
- **[DESCRIPCIÓN]**: Después de firmar, la `PrivateKey` permanece en memoria. Java no garantiza limpieza inmediata de objetos.
- **[ARCHIVO]**: `GenerarContratoAdhesionUseCase.java:164-189`
- **[IMPACTO]**: La clave privada podría permanecer en memoria más tiempo del necesario, aumentando la ventana de exposición si hay un memory dump.
- **[RECOMENDACIÓN]**: Usar `crypto.getInstance().clear()` o implementar un wrapper que haga null la referencia después de usar.

---

### 11. **[MEDIA]**: Uso de `RuntimeException` en lugar de excepciones específicas

- **[CATEGORÍA]**: Estándar
- **[DESCRIPCIÓN]**: El método `calcularHashSha256()` lanza `RuntimeException("SHA-256 not available")` cuando debería lanzar una excepción más específica del dominio.
- **[ARCHIVO]**: Todos los UseCases (ej. `GenerarContratoAdhesionUseCase.java:155-157`)
- **[IMPACTO]**: El catch global de Exception en los UseCases podría capturar esto inapropiadamente.
- **[RECOMENDACIÓN]**: Crear `HashingException` en `domain/exception/` y usarla en lugar de `RuntimeException`.

## CONCLUSIONES

2. **Doble mecanismo de firma digital**: Hay dos implementaciones冲突 (keystore vs Base64) que deben unificarse.

5. **Compliance LOPDP/SUDEBAN**: Falta validar enmascaramiento de datos sensibles en PDFs.

**Recomendación final**: NO desplegar a producción hasta corregir las 4 vulnerabilidades críticas y las 5 altas.
