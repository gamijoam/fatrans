## VIOLACIONES CRÍTICAS (🔴)

### 1. Typo en Endpoint de Desembolso

**Archivo:** `CreditoController.java:215`  
**Severidad:** CRÍTICA  
**Categoría:** A01:2021 Broken Access Control

**Descripción:**
El endpoint `/creditos/solicitudes/{numeroSolicitud}/desembolson` contiene un typo ("desembolson" en lugar de "desembolso"). Esto hace que el endpoint sea inaccesible por la ruta correcta.

**Impacto:**
- El flujo de desembolso de crédito queda completamente roto
- No se puede completar el ciclo de aprobación → desembolso
- Operadores no pueden ejecutar desembolses

**Código actual:**
```java
@PostMapping("/creditos/solicitudes/{numeroSolicitud}/desembolson")
```

**Corrección:**
```java
@PostMapping("/creditos/solicitudes/{numeroSolicitud}/desembolso")
```

### 2. Colateral No Obligatorio en Aprobación

**Archivo:** `AprobarSolicitudCreditoUseCase.java:58-71`  
**Severidad:** CRÍTICA  
**Categoría:** A04:2021 Insecure Design

**Descripción:**
Cuando `solicitud.getColateralCuentaId() == null`, el colateral requerido se calcula como `BigDecimal.ZERO` y la solicitud se aprueba sin verificar ninguna garantía. Esto es un BYPASS de la regla de negocio que requiere colateral para ciertos tipos de crédito.

**Impacto:**
- Créditos que requieren colateral pueden ser aprobados sin él
- Pérdida financiera potencial por falta de garantía
- Violación de reglas de negocio FinTech

**Código problemático:**
```java
BigDecimal colateralRequerido = solicitud.getMontoSolicitado()
    .multiply(tipoCredito.getPorcentajeRequerimientoColateral());

if (solicitud.getColateralCuentaId() != null) {
    // Solo aquí se valida el colateral
    boolean tieneSaldoSuficiente = cuentaGarantiaRepository
        .verificarSaldoParaColateral(solicitud.getColateralCuentaId(), colateralRequerido);
    // ...
} // Si es null, se aprueba sin colateral!
```

**Corrección:**
```java
// Validar colateral REQUERIDO
if (tipoCredito.getPorcentajeRequerimientoColateral()
        .compareTo(BigDecimal.ZERO) > 0) {
    // Colateral es requerido para este tipo de crédito
    if (solicitud.getColateralCuentaId() == null) {
        throw new ColateralRequeridoException(tipoCredito.getNombre());
    }
    boolean tieneSaldoSuficiente = cuentaGarantiaRepository
        .verificarSaldoParaColateral(solicitud.getColateralCuentaId(), colateralRequerido);
    if (!tieneSaldoSuficiente) {
        throw new ColateralInsuficienteException(colateralRequerido, ...);
    }
    cuentaGarantiaRepository.retenerSaldo(...);
}
```

## VIOLACIONES ALTAS (🟠)

### 3. Entidades de Dominio Mutables

**Archivos:**
- `domain/model/SolicitudCredito.java`
- `domain/model/Amortizacion.java`
- `domain/model/PlanAmortizacion.java`
- `domain/model/EvaluacionCrediticia.java`

**Severidad:** ALTA  
**Categoría:** Arquitectura Limpia

**Descripción:**
Las entidades de dominio usan `@Setter` de Lombok, permitiendo mutación directa de estado. Esto viola el principio de inmutabilidad del dominio en DDD.

**Impacto:**
- Estado puede ser modificado fuera de los métodos de negocio
- Violación de transiciones de estado controladas
- Dificulta auditing y trazabilidad

**Corrección:**
Reemplazar `@Setter` con setters privados o usar `record` para valores inmutables:

```java
// En lugar de @Setter, usar métodos package-private
void setEstado(EstadoSolicitud estado) {
    this.estado = estado;  // Solo accesible dentro del paquete
}
```

### 4. Race Condition en Verificación de Crédito Activo

**Archivo:** `CrearSolicitudCreditoUseCase.java:41`  
**Severidad:** ALTA  
**Categoría:** A04:2021 Insecure Design

**Descripción:**
`existeCreditoActivoPorSocio()` se verifica sin lock ni transacción atómica. Dos solicitudes concurrentes del mismo socio podrían pasar la validación simultáneamente.

**Impacto:**
- Un socio podría obtener dos créditos activos simultáneamente
- Violación de regla de negocio "un socio, un crédito activo"

**Corrección:**
```java
@Transactional
public SolicitudCreditoResponse ejecutar(...) {
    // Usar lock pessimistic o constraint de BD
    if (solicitudRepository.existeCreditoActivoPorSocioWithLock(socioId)) {
        throw new CreditoActivoExistenteException(socioId);
    }
    // ...
}
```

O agregar `UNIQUE INDEX` en la BD:
```sql
CREATE UNIQUE INDEX idx_solicitud_activa_socio 
ON solicitudes_credito(socio_id) WHERE estado = 'DESEMBOLSADO';
```

### 5. Validación de Capacidad de Pago Inexistente

**Archivo:** `EvaluarSolicitudUseCase.java`  
**Severidad:** ALTA  
**Categoría:** A04:2021 Insecure Design

**Descripción:**
El caso de uso recibe `salarioEstimado` en el request pero **nunca lo usa** para validar que la cuota no exceda un porcentaje razonable (ej: 30% del salario).

**Impacto:**
- Pueden aprobarse créditos que el socio no puede pagar
- Default alto probable
- Pérdida financiera

**Corrección:**
```java
// Después de calcular la cuota
BigDecimal cuotaEstimada = PlanAmortizacion.calcularCuotaFrances(
    solicitud.getMontoSolicitado(),
    tipoCredito.getTasaInteresAnual(),
    solicitud.getPlazoMeses()
);

BigDecimal porcentajeComprometido = cuotaEstimada.divide(
    request.getSalarioEstimado(), 4, RoundingMode.HALF_UP);

if (porcentajeComprometido.compareTo(new BigDecimal("0.30")) > 0) {
    throw new CapacidadPagoInsuficienteException(
        "Cuota compromete " + porcentajeComprometido.multiply(new BigDecimal("100")) + "% del salario");
}
```

### 6. Tasa de Interés Override Sin Límites

**Archivo:** `AprobarRechazarRequest.java:20`  
**Severidad:** ALTA  
**Categoría:** A02:2021 Cryptographic Failures

**Descripción:**
`TasaInteresOverride` no tiene validación de rango. Un ADMIN podría establecer cualquier tasa (incluyendo 0% o 1000%).

**Impacto:**
- Manipulación de tasas para beneficiar/de perjudicar socios
- FRAUD con implicaciones legales y financieras

**Corrección:**
```java
@DecimalMin(value = "0.001", message = "Tasa mínima 0.1%")
@DecimalMax(value = "1.0", message = "Tasa máxima 100%")
private BigDecimal tasaInteresOverride;
```

---

### 7. Rate Limit con Memory Store

**Archivo:** `SimulacionRateLimitFilter.java:35`  
**Severidad:** ALTA  
**Categoría:** A05:2021 Security Misconfiguration

**Descripción:**
`Map<String, Bucket> BUCKETS` es un `ConcurrentHashMap` estático en memoria. Se reinicia con cada despliegue y puede causar OutOfMemory con muchos IPs distintos.

**Impacto:**
- Rate limit bypass tras restart
- Potential memory leak con muchos IPs únicos
- No funciona en entornos con múltiples instancias

**Corrección:**
Usar Redis para分布式 rate limiting:
```java
private final RedisBucket4jProxy redisProxy;  // Bucket4j con Redis backend
```
