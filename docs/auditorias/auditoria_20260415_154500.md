## VIOLACIONES ALTAS (🟠)

### 1. Transacción Extendida en Batch Sin Límite de Elementos

**Archivo:** `application/usecase/CalcularRendimientosBatchUseCase.java:40-128`  
**Severidad:** ALTA  
**Categoría:** A04:2021 Insecure Design

**Descripción:**
El método `ejecutar()` tiene `@Transactional` que abarca TODO el loop de procesamiento:

```java
@Transactional
public CalcularBatchResponse ejecutar(CalcularBatchRequest request, UUID socioIdToken, boolean isAdmin) {
    // ...
    for (UUID cuentaId : request.getCuentaIds()) {  // <-- Transacción cubre todo el loop
        // ... procesamiento ...
    }
    // ...
}
```

Si `request.getCuentaIds()` contiene miles de elementos, la transacción podría:
- Mantener locks en la BD por tiempo prolongado
- Causing rollback masivo si falla al final
- Exceder timeout de transacción

**Impacto:**
- Degradación de rendimiento en entornos con múltiples instancias
- Potencial deadlock o timeout de BD
- Dificultad para hacer rollback selectivo

**Corrección:**
```java
@Transactional
public CalcularBatchResponse ejecutar(CalcularBatchRequest request, UUID socioIdToken, boolean isAdmin) {
    int totalCuentas = request.getCuentaIds().size();
    
    // Validar límite de batch
    if (totalCuentas > 1000) {
        throw new IllegalArgumentException("Batch excede límite máximo de 1000 cuentas");
    }
    
    // Procesar con chunking
    for (List<UUID> chunk : Lists.partition(request.getCuentaIds(), 100)) {
        procesarChunk(chunk);
    }
    // ...
}

private void procesarChunk(List<UUID> cuentaIdsChunk) {
    // Cada chunk es su propia transacción
}
```

### 2. Falta Validación de Formato UUID en DTOs Públicos

**Archivo:** `application/dto/CalcularBatchRequest.java:33`  
**Severidad:** ALTA  
**Categoría:** A08:2021 Software and Data Integrity Failures

**Descripción:**
El DTO `CalcularBatchRequest` acepta `List<UUID> cuentaIds` sin validación personalizada:

```java
@NotEmpty(message = "cuentaIds no puede estar vacío")
private List<UUID> cuentaIds;
```

Si se envía un UUID malformado (ej: "not-a-uuid"), Spring/JPA lanzarña un `MethodArgumentTypeMismatchException` genérico, no un mensaje amigable.

**Impacto:**
- Mensaje de error técnico expuesto en producción
- Experiencia de usuario deficiente en API
- Potential Information Disclosure

**Corrección:**
```java
@Data
@NoArgsConstructor
@AllArgsConstructor
public class CalcularBatchRequest {

    @NotNull(message = "periodoInicio es requerido")
    private LocalDate periodoInicio;

    @NotNull(message = "periodoFin es requerido")
    private LocalDate periodoFin;

    @NotNull(message = "tipo es requerido")
    private TipoRendimiento tipo;

    @NotEmpty(message = "cuentaIds no puede estar vacío")
    @Size(max = 1000, message = "Máximo 1000 cuentas por batch")
    private List<UUID> cuentaIds;
}
```

Y en el exception handler:
```java
@ExceptionHandler(MethodArgumentTypeMismatchException.class)
public ResponseEntity<ErrorResponse> handleTypeMismatch(MethodArgumentTypeMismatchException ex) {
    if (ex.getRequiredType() == UUID.class) {
        return ResponseEntity.badRequest().body(
            new ErrorResponse("INVALID_UUID", 
                "Formato UUID inválido para parámetro: " + ex.getName())
        );
    }
    // ...
}
```

## RESPUESTAS A PREGUNTAS

### 1. ¿La vulnerabilidad UUID→Long está completamente corregida?

**SÍ.** El análisis confirma:
- `CuentaAhorroEntity` ahora usa `@Id @GeneratedValue(strategy = GenerationType.UUID)`
- `CuentaAhorroJpaRepository` extiende correctamente `JpaRepository<CuentaAhorroEntity, UUID>`
- `CuentaGarantiaRepositoryImpl` **eliminó completamente** el método `uuidToLong()`
- Todos los accesos a cuentas usan `UUID` directamente sin conversión

### 2. ¿Hay nuevos riesgos introducidos por la migración?

**NO hay riesgos CRÍTICOS**, pero sí dos consideraciones de diseño:

1. **Transacción extendida**: El `@Transactional` en `CalcularRendimientosBatchUseCase` abarca todo el loop - podría serproblemático con batches grandes
2. **Validación UUID**: No hay mensaje de error personalizado para UUIDs malformados en endpoints públicos

### 3. ¿El módulo AHORROS está ahora seguro para producción?

**SÍ.** El módulo está listo para producción porque:

| Factor | Estado |
|--------|--------|
| Vulnerabilidad UUID→Long | ✅ Corregida |
| IDOR Checks | ✅ Implementados en todos los endpoints |
| Validación de estado decuenta | ✅ RN-005 verificado |
| Límite de depósito (500k MXN) | ✅ Implementado |
| Límite de retiro diario (50k MXN) | ✅ Implementado |
| Rate limiting | ✅ Implementado |
| Transacciones | ✅ @Transactional en operaciones de escritura |

### 4. ¿Qué validaciones adicionales recomienda?

**Recomendaciones opcionales:**

1. **Limitar tamaño de batch** a 1000元素 para evitar transacciones largas
2. **Personalizar mensajes de error** para UUIDs malformados
3. **Considerar chunking** si el volumen de cuentas es muy alto
4. **Agregar métrica** de duración de transacciones batch
