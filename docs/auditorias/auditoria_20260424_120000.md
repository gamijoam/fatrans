## VIOLACIONES CRÍTICAS (🔴)

### 1. [CRÍTICA] Validación de JWT Client-Side - Falta de Firma y Verificación

**Archivos afectados:**
- `frontend-web/src/app/api/admin/creditos/solicitudes/route.ts:7-16`
- `frontend-web/src/app/api/admin/creditos/solicitudes/[numero]/route.ts:7-16`
- `frontend-web/src/app/api/admin/creditos/solicitudes/[numero]/evaluar/route.ts:7-16`
- `frontend-web/src/app/api/admin/creditos/solicitudes/[numero]/aprobar/route.ts:7-16`
- `frontend-web/src/app/api/admin/creditos/solicitudes/[numero]/rechazar/route.ts:7-16`
- `frontend-web/src/app/api/admin/creditos/solicitudes/[numero]/desembolsar/route.ts:7-16`
- `frontend-web/src/app/api/admin/creditos/solicitudes/[numero]/plan/route.ts:7-16`

**Descripción:** La función `validateAdminRole` decodifica el JWT en el frontend sin verificar la firma cryptográfica:

```typescript
function validateAdminRole(token: string): boolean {
  try {
    const payload = token.split('.')[1];  // Extracción client-side
    const decoded = Buffer.from(payload, 'base64').toString('utf-8');  // Sin verificación de firma
    const data = JSON.parse(decoded);
    return ADMIN_ROLES.includes(data.rol);
  } catch {
    return false;
  }
}
```

**Impacto:** Un atacante puede manipular el JWT en el navegador (via DevTools o script malicioso), cambiar el rol a "ADMIN", y obtener acceso completo a todas las operaciones administrativas de crédito.

**Corrección:** Eliminar esta validación client-side. El backend debe:
1. Validar y verificar la firma del JWT
2. Retornar 401/403 según corresponda
3. El frontend solo debe pasar el token y manejar respuestas de error

### 2. [CRÍTICA] Sin Validación de Inputs en Body POST - Posible Inyección

**Archivos afectados:**
- `frontend-web/src/app/api/admin/creditos/solicitudes/[numero]/evaluar/route.ts:33`
- `frontend-web/src/app/api/admin/creditos/solicitudes/[numero]/aprobar/route.ts:33`
- `frontend-web/src/app/api/admin/creditos/solicitudes/[numero]/rechazar/route.ts:33`
- `frontend-web/src/app/api/admin/creditos/solicitudes/[numero]/desembolsar/route.ts:33`

**Descripción:** Los body JSON se pasan directamente al backend sin validación ni sanitización:

```typescript
const body = await request.json();  // Sin validación de esquema
```

**Impacto:** Datos maliciosos o malformados llegan al backend. Aunque el backend debería validar, la defensa en profundidad requiere validación en ambas capas.

**Corrección:** Validar el body con Zod antes de enviar:

```typescript
import { z } from 'zod';

const EvaluarSchema = z.object({
  puntajeAntiguedad: z.number().min(0).max(30),
  puntajeHistorialAhorro: z.number().min(0).max(30),
  puntajeCapacidadPago: z.number().min(0).max(40),
  salarioEstimado: z.number().min(0),
});
```

## VIOLACIONES DE ARQUITECTURA (🟠)

### 3. [ALTA] IDOR - Sin Verificación de Acceso a Solicitud

**Archivos afectados:**
- `frontend-web/src/app/api/admin/creditos/solicitudes/[numero]/route.ts:32-35`
- `frontend-web/src/app/api/admin/creditos/solicitudes/[numero]/evaluar/route.ts:32-36`
- `frontend-web/src/app/api/admin/creditos/solicitudes/[numero]/aprobar/route.ts:32-36`
- `frontend-web/src/app/api/admin/creditos/solicitudes/[numero]/rechazar/route.ts:32-36`
- `frontend-web/src/app/api/admin/creditos/solicitudes/[numero]/desembolsar/route.ts:32-36`
- `frontend-web/src/app/api/admin/creditos/solicitudes/[numero]/plan/route.ts:32-35`

**Descripción:** El `numero` de solicitud se toma de los params y se pasa directamente al backend sin verificar si el admin autenticado tiene acceso a esa solicitud específica.

**Impacto:** Un admin podría acceder/modificar solicitudes de otros admins o de otras ramas si conoce el número de solicitud.

**Corrección:** Validar formato del `numero`:

```typescript
const { numero } = await params;
if (!numero || !/^[A-Z0-9-]+$/.test(numero)) {
  return NextResponse.json({ message: 'Parámetro inválido' }, { status: 400 });
}
```

### 4. [ALTA] XSS Potencial - Datos del Backend Sin Sanitizar

**Archivos afectados:**
- `frontend-web/src/app/(admin)/admin/creditos/page.tsx:267-276, 307`
- `frontend-web/src/app/(admin)/admin/creditos/[numero]/page.tsx:307, 364, 409`

**Descripción:** Datos del backend se renderizan directamente. Si el backend no sanitiza outputs, hay riesgo XSS:

```typescript
// page.tsx línea 267 - socioNombre viene del backend
<div className="font-medium">{solicitud.socioNombre}</div>

// page.tsx línea 307 - destinoCredito podría contener script
<p className="font-medium">{solicitud.destinoCredito}</p>
```

**Impacto:** Si el backend almacena datos manipulados, el XSS ejecutará en el navegador del admin.

**Corrección:** Verificar que no hay uso de dangerouslySetInnerHTML con datos del backend.

### 5. [ALTA] Manejo de Errores de JSON Parsing Inconsistente

**Archivos afectados:**
- `frontend-web/src/app/api/admin/creditos/solicitudes/[numero]/evaluar/route.ts:59-65`
- `frontend-web/src/app/api/admin/creditos/solicitudes/[numero]/aprobar/route.ts:59-65`
- `frontend-web/src/app/api/admin/creditos/solicitudes/[numero]/rechazar/route.ts:59-65`
- `frontend-web/src/app/api/admin/creditos/solicitudes/[numero]/desembolsar/route.ts:59-65`

**Descripción:** El catch global solo retorna 500, pero request.json() puede lanzar SyntaxError si el body no es JSON válido.

**Impacto:** Un body malformado retorna 500 genérico en lugar de 400 con mensaje específico.

**Corrección:** Separar el parsing JSON del catch global.

### 6. [ALTA] Código Duplicado - validateAdminRole Repetido 7 Veces

**Archivos afectados:** Todos los archivos de ruta API.

**Descripción:** La función validateAdminRole está copiada literalmente en 7 archivos diferentes.

**Impacto:** Violación DRY, mantenibilidad dificultosa, violates Clean Architecture.

**Corrección:** Crear módulo compartido en `frontend-web/src/lib/auth/validate-admin.ts`.
