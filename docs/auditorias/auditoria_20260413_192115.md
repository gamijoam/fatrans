## 2. VIOLACIONES CRÍTICAS

### 🔴 BUG CRÍTICO #1: Normalización Inefectiva en JwtProperties

**Archivo:** `com/tufondo/auth/infrastructure/configuration/JwtProperties.java`  
**Líneas:** 36-38 y 45-47 (compact constructors anidados)

**Problema:** La normalización con `Math.abs()` en los compact constructors de `AccessTokenConfig` y `RefreshTokenConfig` **NO modifica** los valores de los parámetros porque en Java los parámetros de records son `final`.

```java
// ❌ CÓDIGO ACTUAL (Líneas 31-47)
public record AccessTokenConfig(int expirationMinutes) {
    public AccessTokenConfig {
        if (expirationMinutes <= 0) {
            throw new IllegalStateException(
                "jwt.accessToken.expirationMinutes debe ser un número positivo");
        }
        // ⚠️ BUG: Math.abs() aquí NO tiene efecto - expirationMinutes es final
        expirationMinutes = Math.abs(expirationMinutes);
    }
}

public record RefreshTokenConfig(int expirationDays) {
    public RefreshTokenConfig {
        if (expirationDays <= 0) {
            throw new IllegalStateException(
                "jwt.refreshToken.expirationDays debe ser un número positivo");
        }
        // ⚠️ BUG: Math.abs() aquí NO tiene efecto - expirationDays es final
        expirationDays = Math.abs(expirationDays);
    }
}
```

**Impacto:** Si alguien configura `jwt.accessToken.expirationMinutes=-60`, el código:
1. Verifica que es ≤ 0 ✅
2. Lanza `IllegalStateException` ✅
3. **Nunca ejecuta** la línea del `Math.abs()` porque la excepción ya fue lanzada

Esto convierte el comentario "Normalizar a positivo" en código muerto.

**Corrección requerida:**
```java
// ✅ CÓDIGO CORRECTO
public record AccessTokenConfig(int expirationMinutes) {
    public AccessTokenConfig {
        if (expirationMinutes <= 0) {
            throw new IllegalStateException(
                "jwt.accessToken.expirationMinutes debe ser un número positivo");
        }
        // Ya que es > 0, no necesita normalización.
        // Si quisieras aceptar negativos, usarías una variable temporal:
        // int normalized = Math.abs(expirationMinutes);
    }
}
```

**Recomendación:** Eliminar los `Math.abs()` engañosos o clarificar que son intencionalmente código muerto (documentación).

### 🔴 BUG CRÍTICO #2: Reasignación Innecesaria en Record Anidado

**Archivo:** `com/tufondo/auth/infrastructure/configuration/JwtProperties.java`  
**Líneas:** 27-30

**Problema:** En el compact constructor principal se reasignan `accessToken` y `refreshToken` creando nuevos records, pero esto es innecesario y potencialmente confuso.

```java
// ❌ CÓDIGO ACTUAL (Líneas 27-30)
public JwtProperties {
    Objects.requireNonNull(accessToken, ...);
    Objects.requireNonNull(refreshToken, ...);
    
    // ⚠️ REASIGNACIÓN INNECESARIA
    accessToken = new AccessTokenConfig(
        Objects.requireNonNull(accessToken.expirationMinutes(), ...)
    );
    refreshToken = new RefreshTokenConfig(
        Objects.requireNonNull(refreshToken.expirationDays(), ...)
    );
}
```

**Impacto:** El código ya validó que `accessToken` no es null con `Objects.requireNonNull(accessToken, ...)`. Crear un nuevo `AccessTokenConfig` solo para validar la propiedad anidada es redundante. Los objetos ya creados por Spring Boot@ConfigurationProperties ya tienen los valores configurados.

**Corrección requerida:**
```java
// ✅ CÓDIGO CORRECTO
public JwtProperties {
    Objects.requireNonNull(secret, "jwt.secret es requerido...");
    Objects.requireNonNull(accessToken, "jwt.accessToken es requerido...");
    Objects.requireNonNull(refreshToken, "jwt.refreshToken es requerido...");
    Objects.requireNonNull(issuer, "jwt.issuer es requerido...");
    
    // La validación anidada se delega al compact constructor de cada record
}
```

### ⚡ MEJORA #1: Inconsistencia en Normalización de Correo

**Archivos:**
- `com/tufondo/auth/infrastructure/persistence/adapter/UsuarioRepositoryImpl.java` (líneas 32-35)
- `com/tufondo/auth/infrastructure/persistence/jpa/UsuarioJpaRepository.java`

**Descripción:** La normalización `toLowerCase().trim()` se aplica en el adapter pero no está documentada ni es consistente con todos los métodos de búsqueda.

```java

return jpaRepository.findByCorreoElectronicoAndCuentaActivaTrue(correo.toLowerCase().trim())
```

**Riesgo:** Si un desarrollador usa directamente el JpaRepository sin normalizar, obtendrá resultados inconsistentes.

**Recomendación:** Agregar un comentario Javadoc en el JpaRepository indicando que se requiere normalización previa, o implementar normalización automática usando `@PrePersist`/`@PreUpdate`.
