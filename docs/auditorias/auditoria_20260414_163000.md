## 🔴 VIOLACIONES CRÍTICAS

### 1. **[CRÍTICA]** - AUDITORÍA KYC NO IMPLEMENTADA (TABLA SHADOW)

- **[CATEGORÍA]**: Seguridad / Cumplimiento Regulatorio
- **[DESCRIPCIÓN]**: La tabla `audit_kyc` está definida en el modelo de datos (MODELO_DATOS.md líneas 156-177) y la API.md especifica códigos de error KYC_XXX para auditoría, pero **no existe ninguna implementación** de logging de auditoría en el código. No hay:
  - Entity `AuditKYCEntity`
  - Repository `AuditKYCRepository`
  - Uso de `@PreAuthorize` con logging de accesos
  - Interceptores o aspectos para capturar eventos de auditoría

- **[IMPACTO]**: **VIOLACIÓN CRÍTICA DE LOPDP y SUDEBAN**. En caso de auditoría regulatoria, no habrá trazabilidad de:
  - Quién accedió a cada documento de identidad
  - Qué acciones realizó cada analista
  - Intentos de acceso no autorizado
  - Cambios de estado en verificaciones KYC

- **[RECOMENDACIÓN]**: Implementar `AuditKYCEntity` + `AuditKYCAuditorListener` usando JPA `@PrePersist`/`@PreUpdate` o un AspectJ interceptor que capture automáticamente todas las operaciones CRUD sobre `VerificacionKYC`, `DocumentoIdentidad` y `ConsentimientoKYC`.

---

### 2. **[CRÍTICA]** - RATE LIMITING NO IMPLEMENTADO

- **[CATEGORÍA]**: Seguridad / OWASP C5
- **[DESCRIPCIÓN]**: La API.md (líneas 690-699) especifica:
  - 60 requests/min por IP (default)
  - 20 requests/min para `/kyc/documentos` por usuario
  - 30 requests/min para `/kyc/cola-revision` por analista
  - 100 MB/socio/día límite de almacenamiento
  - 3 verificaciones máximas por socio/día
  
  **No existe ninguna implementación** de estos límites. Búsqueda con grep de `@RateLimiter`, `RateLimiter`, `rate.limit` retorna vacío.

- **[IMPACTO]**: Un atacante puede realizar:
  - Ataques de fuerza bruta sobre endpoints KYC
  - Inundación de uploads (DoS por storage)
  - Abuso del servicio sin detección

- **[RECOMENDACIÓN]**: Implementar usando Bucket4j o Resilience4j:
```java
// Ejemplo de implementación
@Bean
public FilterRegistrationBean<RateLimitingFilter> rateLimiterFilter() {
    // Configurar límites según API.md líneas 690-699
}
```

### 3. **[CRÍTICA]** - IDOR (Insecure Direct Object Reference) EN ANALISTA CONTROLLER

- **[CATEGORÍA]**: Seguridad / OWASP A01
- **[DESCRIPCIÓN]**: En `AnalistaKYCController.java:37-44`:
```java
@GetMapping("/revision/{verificacionId}")
@PreAuthorize("hasAnyRole('ANALISTA_KYC', 'ADMIN')")
public ResponseEntity<RevisionResponse> obtenerDetalleRevision(
        @PathVariable UUID verificacionId) {
    RevisionResponse response = revisarDocumentosUseCase.obtenerDetalle(verificacionId);
```
**NO hay validación** de que:
1. La verificación esté en estado `EN_REVISION`
2. El analista tenga asignada esa verificación (segregación de trabajo)

- **[IMPACTO]**: Un analista malicioso puede acceder a cualquier verificación KYC por su ID, incluso si no está en su cola de trabajo. Expone datos personales de socios sin autorización.

- **[RECOMENDACIÓN]**: Agregar validación en `RevisarDocumentosUseCase.obtenerDetalle()`:
```java
VerificacionKYC verificacion = verificacionRepository.findById(verificacionId)
    .orElseThrow(...);

if (verificacion.getEstado() != EstadoVerificacion.EN_REVISION) {
    throw new AccesoNoAutorizadoException("Verificacion no disponible para revision");
}
```

### 4. **[CRÍTICA]** - SIN OPTIMISTIC LOCKING EN ENTIDADES JPA

- **[CATEGORÍA]**: Concurrencia / Integridad de Datos
- **[DESCRIPCIÓN]**: Las entidades JPA `VerificacionKYCEntity.java` y `DocumentoIdentidadEntity.java` **NO tienen `@Version`**. Búsqueda de `@Version` retorna vacío en todo el módulo.

- **[IMPACTO]**: Condiciones de carrera cuando:
  - Múltiples analistas intentan aprobar/rechazar simultáneamente
  - Un socio sube documentos mientras el sistema valida otros
  - Transiciones de estado concurrentes (RECHAZADO → PENDIENTE → EN_REVISION)

- **[RECOMENDACIÓN]**: Agregar `@Version` a ambas entidades:
```java
@Version
@Column(name = "version")
private Long version;
```

## 🟠 VIOLACIONES DE ARQUITECTURA / ALTAS

### 5. **[ALTA]** - STORAGE URLs EXPUESTAS EN REVISIÓN (PATH TRAVERSAL POTENCIAL)

- **[CATEGORÍA]**: Seguridad / OWASP A01, A03
- **[DESCRIPCIÓN]**: En `RevisionResponse.java:43` se expone `urlVisualizacion` que es la URL completa de MinIO:
```java
String urlPresignada = storagePort.generatePresignedUrl(doc.getUrlAlmacenamiento(), 15);
```
El `urlAlmacenamiento` en `DocumentoIdentidadEntity` se almacena como `bucketName + "/" + fullPath` (MinIOStorageService.java:60), lo que podría revelar la estructura interna del storage.

- **[IMPACTO]**: Si un atacante obtiene URLs presignadas, podría inferir la estructura de directorios y potencialmente realizar ataques de path traversal si la sanitización en `normalizePath()` es insuficiente.

- **[RECOMENDACIÓN]**: 
  1. No exponer `urlAlmacenamiento` internamente, solo la URL presignada final
  2. Verificar que `normalizePath()` elimina todos los caracteres especiales

### 6. **[ALTA]** - VERSION POLITICA SIN VALIDACIÓN

- **[CATEGORÍA]**: Seguridad / OWASP A05
- **[DESCRIPCIÓN]**: En `IniciarKYCRequest.java:24-25` y `IniciarKYCUseCase.java:50`:
```java
@NotNull(message = "versionPolitica es requerido")
private String versionPolitica;
...

```
Se acepta **cualquier string** como versión de política sin consultar una tabla `politica_privacidad` activa.

- **[IMPACTO]**: Un atacante podría registrar consentimientos con versiones de política falsificadas, creando registros de compliance inválidos.

- **[RECOMENDACIÓN]**: Validar contra una tabla `politica_privacidad`:

```java
boolean versionValida = politicaPrivacidadRepository
    .existsByVersionAndActivaTrue(request.getVersionPolitica());
if (!versionValida) {
    throw new PoliticaPrivacidadInvalidaException();
}
```

### 7. **[ALTA]** - FALTA ENDPOINT DE REVOCACIÓN DE CONSENTIMIENTO (LOPDP Art. 7)

- **[CATEGORÍA]**: Cumplimiento / LOPDP
- **[DESCRIPCIÓN]**: Según la SPEC.md (línea 1743-1746) y modelo de datos (línea 132-149), existe la tabla `consentimiento_kyc` pero **NO existe**:
  - `RevocarConsentimientoUseCase`
  - Endpoint `POST /kyc/revocar-consentimiento`
  - Lógica para invalidar consentimientos

- **[IMPACTO]**: **VIOLACIÓN DIRECTA DEL ARTÍCULO 7 DE LOPDP** (Derecho de revocación del consentimiento). Los titulares de datos no pueden ejercer su derecho a revocar el consentimiento para el tratamiento de sus datos.

- **[RECOMENDACIÓN]**: Implementar como especifica la SPEC.md líneas 1709-1745:
```java
@PostMapping("/revocar-consentimiento")
@PreAuthorize("hasRole('SOCIO')")
public ResponseEntity<RevocarConsentimientoResponse> revocarConsentimiento(...)
```

### 8. **[ALTA]** - CONSENTIMIENTO NO VALIDA IP NI USER AGENT DEL PROXY

- **[CATEGORÍA]**: Seguridad / Trazabilidad
- **[DESCRIPCIÓN]**: En `KYCController.java:52-58`:
```java
if (request.getIpCliente() == null) {
    request.setIpCliente(getClientIp(httpRequest));
}
```
Se usa `X-Forwarded-For` pero **NO se valida** que la IP sea una IP válida (formato IPv4/IPv6). No se registra si vino a través de proxy/VPN.

- **[IMPACTO]**: Registro de consentimiento puede ser manipulado si el header `X-Forwarded-For` es spoofed. No hay forma de detectar IPs de proxy anónimo.

- **[RECOMENDACIÓN]**: Validar formato IP y registrar si fue obtenido de proxy:
```java
private String getClientIp(HttpServletRequest request) {
    String xForwardedFor = request.getHeader("X-Forwarded-For");
    if (xForwardedFor != null && isValidIp(xForwardedFor.split(",")[0].trim())) {
        return xForwardedFor.split(",")[0].trim();
    }
    return request.getRemoteAddr();
}
```

### 9. **[ALTA]** - VALIDACIÓN DE NOMBRE DE ARCHIVO INSUFICIENTE

- **[CATEGORÍA]**: Seguridad / OWASP A03, A05
- **[DESCRIPCIÓN]**: En `SubirDocumentoRequest.java:31-32`:
```java
@NotBlank(message = "nombreOriginal es requerido")
private String nombreOriginal;
```
Solo se valida que no esté en blanco, **NO se valida**:
- Longitud máxima (255 caracteres según modelo de datos línea 229)
- Caracteres especiales o path traversal (`../`, `..\\`)
- Extensión maliciosa

- **[IMPACTO]**: Podría causar:
  - Buffer overflow si nombreOriginal > 255
  - Path traversal si contiene `../`
  - Inyección en logs o almacenamiento

- **[RECOMENDACIÓN]**: Agregar validaciones:
```java
@NotBlank(message = "nombreOriginal es requerido")
@Size(max = 255, message = "Nombre de archivo muy largo")
@Pattern(regexp = "^[^../\\\\]+$", message = "Nombre de archivo invalido")
private String nombreOriginal;
```
