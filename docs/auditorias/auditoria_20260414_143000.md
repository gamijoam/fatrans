## 2. VIOLACIONES CRÍTICAS (🔴)

### 🔴 CRÍTICO #1: Missing Input Validation en DTOs de Operaciones Financieras

**Categoría:** OWASP A01:2021 - Broken Access Control  
**Severidad:** CRÍTICA

**Descripción:** Las especificaciones NO definen validaciones en los DTOs de entrada para operaciones financieras (depósitos y retiros). Específicamente:

- No se especifica `@DecimalMin` ni `@DecimalMax` para el campo `monto`
- No se define longitud máxima para `referencia` y `descripcion` (potencial injection)
- No se menciona sanitización de `ipOrigen` ni `canalOrigen`

**Impacto:** 
- Un atacante podría enviar montos negativos o extremadamente grandes causando overflow
- Cadenas largas en `referencia` podrían explotar vulnerabilidades de base de datos
- IP spoofing si no se valida formato

**Recomendación:**
```java
// DTO de Depósito
public record DepositoRequestDTO(
    @NotNull @DecimalMin("0.0001") @DecimalMax("500000.00")
    @Digits(integer = 11, fraction = 4)
    BigDecimal monto,
    
    @Size(max = 500) String descripcion,
    
    @Size(max = 100) @Pattern(regexp = "^[A-Z0-9-]+$") String referencia,
    
    @NotNull CanalOperacion canalOrigen
) {}

// DTO de Retiro
public record RetiroRequestDTO(
    @NotNull @DecimalMin("0.0001") @DecimalMax("50000.00")
    @Digits(integer = 11, fraction = 4)
    BigDecimal monto,
    
    @Size(max = 500) String descripcion,
    
    @Size(max = 100) @Pattern(regexp = "^[A-Z0-9-]+$") String referencia,
    
    @NotNull CanalOperacion canalOrigen
) {}
```

### 🔴 CRÍTICO #2: Rate Limiting Insuficiente para Sistema Financiero

**Categoría:** OWASP A04:2021 - Insecure Design  
**Severidad:** CRÍTICA

**Descripción:** Las especificaciones definen:
- 10 req/min para depósitos
- 5 req/min para retiros

Sin embargo, **NO se especifica:**
- Límite global por cuenta (sin importar el canal)
- Límite por IP/sesión
- Window deslizante vs window fijo
- Qué sucede cuando se excede (¿bloqueo temporal? ¿ampliar ventana?)

**Impacto:**
- Un atacante con múltiples cuentas podría realizar ataques de fuerza bruta al sistema
- Límites bajos podrían causar Denial of Service a usuarios legítimos
- No hay protección contra abuse de la API batch

**Recomendación:**
```yaml
rate-limiting:
  depositos:
    por-cuenta: 60  # 1 por segundo
    por-ip: 100
    window-seconds: 60
  retiros:
    por-cuenta: 30
    por-ip: 50
    window-seconds: 60
  batch:
    global-limit: 10
    por-cuenta: 2
```

**Categoría:** OWASP A09:2021 - Security Logging Failures  
**Severidad:** CRÍTICA

**Descripción:** La especificación define eventos de auditoría pero OMITE eventos críticos:

public enum EventoAuditoria {

### 🔴 CRÍTICO #4: Overflow Numérico en Cálculo de Rendimientos (BigDecimal)

**Categoría:** OWASP A04:2021 - Insecure Design  
**Severidad:** CRÍTICA

**Descripción:** La especificación menciona:
- `BigDecimal saldoPromedioPeriodo, tasaAplicada, montoRendimiento`
- `NUMERIC(15,4)` para saldos

**PROBLEMA CRÍTICO:** No se especifica:
1. Validación de rangos para `tasaAplicada` (¿qué pasa si es > 1 para interés mensual simple?)
2. Overflow protection en el cálculo: `montoRendimiento = saldoPromedioPeriodo * tasaAplicada`
3. Límite máximo de rendimiento por período
4. Precisión de cálculo (¿cuántos decimales internos?)

**Impacto:**
- Cálculos podrían generar rendimientos absurdamente grandes
- Underflow podría generar rendimientos de 0.0000 truncados
- Overflow podría causar excepción o wrap-around

**Recomendación:**
```java
public record Rendimiento(
    // ...
) {
    private static final BigDecimal TASA_MAXIMA_MENSUAL = new BigDecimal("0.15"); // 15% mensual max
    private static final BigDecimal MONTO_MAXIMO_RENDIMIENTO = new BigDecimal("999999999.9999");
    private static final int ESCALA_CALCULO = 10;
    
    public static Rendimiento calcular(BigDecimal saldoPromedio, BigDecimal tasa) {
        Objects.requireNonNull(saldoPromedio, "Saldo no puede ser null");
        Objects.requireNonNull(tasa, "Tasa no puede ser null");
        
        if (tasa.compareTo(BigDecimal.ZERO) < 0 || tasa.compareTo(TASA_MAXIMA_MENSUAL) > 0) {
            throw new IllegalArgumentException("Tasa fuera de rango permitido");
        }
        
        BigDecimal rendimiento = saldoPromedio
            .multiply(tasa)
            .setScale(ESCALA_CALCULO, RoundingMode.HALF_UP)
            .setScale(4, RoundingMode.HALF_UP); // Para almacenamiento
        
        if (rendimiento.compareTo(MONTO_MAXIMO_RENDIMIENTO) > 0) {
            throw new ArithmeticException("Rendimiento excede límite máximo");
        }
        
        return new Rendimiento(/* ... */);
    }
}
```

### 🔴 CRÍTICO #5: Falta Validación de Propietario en Todos los Endpoints

**Categoría:** OWASP A01:2021 - Broken Access Control  
**Severidad:** CRÍTICA

**Descripción:** La especificación indica:
> "Autorización: socioId extraido del JWT, no del request body"

**PERO** las especificaciones NO clarifican:
1. ¿Cómo se valida que el `socioId` del JWT tiene acceso a `numeroCuenta` en la URL?
2. ¿Solo el propietario puede ver sus propios movimientos?
3. ¿Un ADMIN puede ver todas las cuentas?
4. ¿Existe verificación de "beneficiary" para transferencias?

**Endpoints afectados:**
- `GET /api/v1/ahorros/cuentas/{numeroCuenta}` - ¿Solo dueño?
- `GET /api/v1/ahorros/cuentas/{numeroCuenta}/saldo` - ¿Solo dueño?
- `GET /api/v1/ahorros/cuentas/{numeroCuenta}/movimientos` - ¿Solo dueño?
- `DELETE /api/v1/ahorros/cuentas/{numeroCuenta}` - ¿Solo dueño?

**Impacto:**
- IDOR (Insecure Direct Object Reference) - Un usuario podría acceder a cuentas de otro
- Violación de privacidad financiera (LOPDP)
- Auditoría regulatoria comprometida

**Recomendación:**
```java
// En cada UseCase, verificar propiedad
public class VerCuentaUseCase {
    public CuentaAhorroResponseDTO ejecutar(String numeroCuenta, UUID socioIdJWT) {
        CuentaAhorro cuenta = cuentaRepository.buscarPorNumero(numeroCuenta)
            .orElseThrow(() -> new CuentaNoEncontradaException(...));
        
        if (!cuenta.getSocioId().equals(socioIdJWT)) {
            auditService.registrar(EventoAuditoria.ACCESO_CUENTA_AJENA, socioIdJWT, numeroCuenta);
            throw new AccesoNoAutorizadoException("AHORROS_012");
        }
        
        return mapper.toDTO(cuenta);
    }
}
```

## 3. VIOLACIONES DE ARQUITECTURA (🟠)

### 🟠 ARQUITECTURA #1: Entidad CuentaAhorro Mezcla Estado con Comportamiento

**Categoría:** Arquitectura - Clean Architecture Violation  
**Severidad:** ALTA

**Descripción:** La entidad `CuentaAhorro` combina:
- Datos (`UUID id, BigDecimal saldoActual, saldoRetenido...`)
- Estado (`EstadoCuenta estado`)
- Reglas de negocio (implícitas en los campos)

**PROBLEMA:** No hay evidencia de Value Objects o métodos de dominio que encapsulen:
- La lógica de transición de estados
- Cálculo de `saldoDisponible`
- Validaciones de operación

**Impacto:**
- Violación de encapsulamiento
- Lógica de negocio dispersa en UseCases
- Dificultad para testing

**Recomendación:**
```java
public final class CuentaAhorro {
    private final UUID id;
    private final Saldo saldo; // Value Object
    private final EstadoCuenta estado;
    private final TipoCuenta tipoCuenta;
    
    public Moneda saldoDisponible() {
        return saldo.disponible();
    }
    
    public ResultadoOperacion puedeDepositar(Moneda monto) {
        if (estado != EstadoCuenta.ACTIVA) {
            return ResultadoOperacion.error("AHORROS_004");
        }
        if (monto.mayorQue(LIMITE_DEPOSITO)) {
            return ResultadoOperacion.error("AHORROS_006");
        }
        return ResultadoOperacion.exitoso();
    }
}
```

### 🟠 ARQUITECTURA #3: Falta Definición de Batch como Contexto Delimitado

**Categoría:** Arquitectura - DDD  
**Severidad:** MEDIA

**Descripción:** El endpoint `POST /api/v1/ahorros/rendimientos/calcular-batch` se especifica como parte del mismo módulo sin un bounded context separado.

**PROBLEMA:**
- Un batch failure podría dejar el sistema en estado inconsistente
- No hay definición de saga/compensación
- ¿Qué sucede si 1 de 1000 falla? ¿Se hace rollback total o parcial?

**Impacto:**
- Procesamiento de errores inconsistente
- Posibles cálculos duplicados
- Dificultad para auditar

**Recomendación:** Definir claramente:
```java
// Command para batch
public record CalcularRendimientosBatchCommand(
    @NotEmpty List<UUID> cuentaIds,
    LocalDate periodoInicio,
    LocalDate periodoFin,
    boolean atomicMode // true = todo o nada, false = mejor esfuerzo
) {}

// Events
public sealed interface RendimientoBatchEvent {
    record Exitoso(UUID cuentaId, Rendimiento rendimiento) implements RendimientoBatchEvent {}
    record Fallido(UUID cuentaId, String motivo) implements RendimientoBatchEvent {}
    record Completado(int exitosos, int fallidos) implements RendimientoBatchEvent {}
}
```

### 🟠 ARQUITECTURA #4: Campos de Auditoría No Estandarizados

**Categoría:** Arquitectura - ISO 27001 / SUDEBAN  
**Severidad:** ALTA

**Descripción:** La especificación menciona "Campos auditoría" pero:
- No hay estandarización de formato
- `creadoPor`, `actualizadoPor` son String vs UUID inconsistente
- No hay `fechaEliminacion` (soft delete)
- No hay `motivoEliminacion` para logs de auditoría

**Impacto:**
- Dificultad para correlación de logs
- Incumplimiento de retención de registros (SUDEBAN exige 10 años mínimo)
- No hay trazabilidad de cambios de estado

**Recomendación:**
```java
public interface AuditMetadata {
    UUID getCreadoPor();
    Instant getFechaCreacion();
    UUID getActualizadoPor();
    Instant getFechaActualizacion();
    boolean getEliminado(); // Soft delete
    UUID getEliminadoPor();
    Instant getFechaEliminacion();
    String getMotivoEliminacion();
}

public record StandardAudit(
    UUID creadoPor,
    Instant fechaCreacion,
    UUID actualizadoPor,
    Instant fechaActualizacion,
    boolean eliminado,
    UUID eliminadoPor,
    Instant fechaEliminacion,
    String motivoEliminacion
) implements AuditMetadata {}
```

## 4. VIOLACIONES DE NEGOCIO (REGLAS DE NEGOCIO)

### ⚠️ INCONSISTENCIA #1: RN-AHO-03 Contradice otras Reglas

**Descripción:** La regla dice:
> "RN-AHO-03: Retiro no puede dejar saldo por debajo de montoMinimoRequerido (warning, no bloqueo)"

**CONTRADICCIÓN con:**
- RN-AHO-07: "No cerrar con saldoRetenido > 0" - sugiere que el `saldoRetenido` NO cuenta como saldo disponible
- RN-AHO-09: "saldoDisponible = saldoActual - saldoRetenido"

**PREGUNTA:** ¿El `montoMinimoRequerido` aplica sobre `saldoActual` o `saldoDisponible`?

**Impacto:** Ambigüedad podría causar:
- Cálculos incorrectos de límite de retiro
- Cláusula de protección al consumidor violada

**Recomendación:** Clarificar:
```java
public record TipoRetiro(
    BigDecimal monto,
    CuentaAhorro cuenta
) {
    public BigDecimal getSaldoDisponible() {
        return cuenta.getSaldoActual().subtract(cuenta.getSaldoRetenido());
    }
    
    public boolean violaMontoMinimo() {
        return getSaldoDisponible().subtract(monto)
            .compareTo(cuenta.getMontoMinimoRequerido()) < 0;
    }
}
```
