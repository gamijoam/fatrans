## VIOLACIONES CRÍTICAS (🔴)

### 1. Rate Limiting Inexistente para Operaciones Financieras

- **[CRITICIDAD]:** 🔴 CRÍTICA
- **[CATEGORÍA]:** Seguridad - OWASP API Security Top 10
- **[DESCRIPCIÓN]:** Las especificaciones declaran rate limits específicos pero el código existente SÓLO implementa rate limiting para `/api/v1/auth/login` (5 req/min). No existe implementación para:
  - 10 depósitos/min
  - 3 retiros/min
  - 30 consultas saldo/min
  - 20 consultas movimientos/min

- **[IMPACTO]:** Un atacante podría realizar ataques de fuerza bruta o amplificación contra las APIs de depósitos y retiros. También expone la lógica de negocio a abuso sin costo financiero inmediato.

- **[RECOMENDACIÓN]:** Implementar `RateLimitFilter` genérico usando Bucket4j (ya disponible como dependencia) con límites por operación:

```java
// En infraestructura/security/RateLimitFilter.java
@Bean
public Map<String, Bucket> operationBuckets() {
    return Map.of(
        "DEPOSITO", BucketFactory.createPorMinuto(10),
        "RETIRO", BucketFactory.createPorMinuto(3),
        "SALDO", BucketFactory.createPorMinuto(30),
        "MOVIMIENTOS", BucketFactory.createPorMinuto(20)
    );
}
```

### 2. Falta de Validación de Propiedad de Cuenta

- **[CRITICIDAD]:** 🔴 CRÍTICA
- **[CATEGORÍA]:** Seguridad - Broken Authorization (OWASP Top 10 #1)
- **[DESCRIPCIÓN]:** Las especificaciones indican que `ROLE_SOCIO` debe tener acceso "solo a su propia cuenta", pero no existe `@PreAuthorize` en ningún controller existente que valide que el `socioId` del token JWT coincida con el `socioId` de la cuenta consultada.

- **[IMPACTO]:** Un usuario con rol ROLE_SOCIO podría consultar, depositar o retirar de cuentas de otros socios mediante ID enumeration attacks.

- **[RECOMENDACIÓN]:** Crear anotación de seguridad personalizada y aplicar en todos los endpoints del módulo Ahorros:

```java
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
@PreAuthorize("@cuentaSeguridadService.validarPropiedadCuenta(#cuentaId, authentication)")
public @interface ValidarPropiedadCuenta {}
```

```java
@Service
public class CuentaSeguridadService {
    public boolean validarPropiedadCuenta(UUID cuentaId, Authentication auth) {
        // Extraer socioId del JWT, buscar cuenta, validar coincidencia
    }
}
```

### 3. Concurrency Risk en Saldos - Dirty Read Possible

- **[CRITICIDAD]:** 🔴 CRÍTICA
- **[CATEGORÍA]:** Arquitectura - Integridad Transaccional
- **[DESCRIPCIÓN]:** La entidad `CuentaAhorro` especifica campos `saldoActual`, `saldoDisponible`, `saldoRetenido` que se modifican en cada operación financiera. Sin optimistic locking ni synchronized blocks, dos transacciones concurrentes podrían generar:

```
Transacción A: Lee saldo = $10,000
Transacción B: Lee saldo = $10,000
Transacción A: Depósito $5,000 → Escribe $15,000
Transacción B: Retiro $3,000 → Escribe $12,000 (OVERWRITE!)
Saldo final: $12,000 (debería ser $17,000)
```

- **[IMPACTO]:** Pérdida de dinero en depósitos o retiros excesivos. Violación directa de consistencia financiera.

- **[RECOMENDACIÓN]:** Implementar con `@Version` en JPA o usar pessimistic locking en queries de movimiento:

```java
@Entity
public class CuentaAhorro {
    @Version
    private Long version; // Optimistic locking
    
    @Lock(LockModeType.PESSIMISTIC_WRITE)
    @Query("SELECT c FROM CuentaAhorro WHERE c.id = :id")
    Optional<CuentaAhorro> findByIdWithLock(@Param("id") UUID id);
}
```

### 4. Hash SHA-256 Sin Sal Para Auditoría de Transacciones

- **[CRITICIDAD]:** 🔴 CRÍTICA
- **[CATEGORÍA]:** Seguridad - Data Integrity
- **[DESCRIPCIÓN]:** Las especificaciones declaran: 
> "Hash SHA-256 para auditoría: `numeroTransaccion + monto + fecha + saldoAnterior`"

Este hash es vulnerable a:
- **Colisiones:** Dos transacciones diferentes podrían producir el mismo hash
- **Rainbow Tables:** Sin sal, es vulnerable a tablas precalculadas
- **Longitud fija:** SHA-256 siempre produce 256 bits, sin información de sal

- **[IMPACTO]:** Un atacante podría modificar transacciones en logs y el hash seguiría siendo "válido" si conoce el valor original. No cumpliría como evidencia forense.

- **[RECOMENDACIÓN]:** Usar HMAC-SHA-256 con clave secreta de auditoría:

```java
public String generarHashAuditoria(String numeroTransaccion, BigDecimal monto, 
                                    LocalDateTime fecha, BigDecimal saldoAnterior) {
    String data = numeroTransaccion + monto.toString() + fecha.toString() + saldoAnterior.toString();
    return HMAC_SHA256(data, auditoriaSecretKey); // Clave única por ambiente
}
```

### 7. Validación de Montos En DTOs No Visible

- **[CRITICIDAD]:** 🟠 ALTA
- **[CATEGORÍA]:** Arquitectura - Input Validation
- **[DESCRIPCIÓN]:** Las especificaciones declaran montos mínimos/máximos para depósitos y retiros, pero en el código existente no hay validación en DTOs de entrada para эти límites:

```
Depósitos: min $1,000 / max $500,000 por operación / $2,000,000 diario
Retiros: min $500 / max $50,000 por operación / $150,000 diario / 3 por día
```

- **[IMPACTO]:** Datos inválidos podrían entrar al sistema sin validación, causando errores en cálculo de saldos o excepciones no manejadas.

- **[RECOMENDACIÓN]:** Crear anotación de validación personalizada:

```java
@Target({ElementType.FIELD, ElementType.PARAMETER})
@Constraint(validatedBy = MontoDepositoValidator.class)
public @interface MontoDepositoValido {}

public class MontoDepositoValidator implements ConstraintValidator<MontoDepositoValido, BigDecimal> {
    @Override
    public boolean isValid(BigDecimal monto, ConstraintValidatorContext ctx) {
        return monto != null 
            && monto.compareTo(new BigDecimal("1000")) >= 0
            && monto.compareTo(new BigDecimal("500000")) <= 0;
    }
}
```

### 8. Estados de Movimiento y Cuenta No Validan Transiciones Válidas

- **[CRITICIDAD]:** 🟠 ALTA
- **[CATEGORÍA]:** Arquitectura - Business Logic Integrity
- **[DESCRIPCIÓN]:** Los enums `EstadoCuenta` y `EstadoMovimiento` declarados no muestran transiciones válidas. Por ejemplo:
- Una cuenta no debería ir de `BLOQUEADA` directamente a `CERRADA` sin pasar por `EN_PROCESO_CIERRE`
- Un movimiento `PENDIENTE` no debería poder ser `CANCELADO` si ya está `COMPLETADO`

- **[IMPACTO]:** Estados inconsistentes podrían generar operaciones inválidas (e.g., depositar a cuenta cancelada).

- **[RECOMENDACIÓN]:** Implementar máquina de estados con Transitions:

```java
public class EstadoCuenta {
    private static final Map<EstadoCuenta, Set<EstadoCuenta>> TRANSICIONES_VALIDAS = Map.of(
        ACTIVA, Set.of(BLOQUEADA, EN_PROCESO_CIERRE),
        BLOQUEADA, Set.of(ACTIVA, EN_PROCESO_CIERRE),
        EN_PROCESO_CIERRE, Set.of(CERRADA, ACTIVA), // Reversión posible
        CERRADA, Set.of(), // Terminal
        CANCELADA, Set.of() // Terminal
    );
    
    public boolean puedeTransitarA(EstadoCuenta nuevoEstado) {
        return TRANSICIONES_VALIDAS.get(this).contains(nuevoEstado);
    }
}
```

### 9. Missing Idempotency Key Para Prevención de Double-Payments

- **[CRITICIDAD]:** 🟠 ALTA
- **[CATEGORÍA]:** Seguridad - Transaction Integrity
- **[DESCRIPCIÓN]:** No existe especificación de `idempotencyKey` en los endpoints de depósito/retiro. Sin esto, reintentos de red podrían causar doble cargo.

- **[IMPACTO]:** Un cliente que presiona "Depositar" dos veces por error de red podría ver два cargos. O un proxy podría reenviar requests.

- **[RECOMENDACIÓN]:** Implementar capa de idempotencia:

```java
@Bean
public IdempotencyService idempotencyService(JdbcTemplate jdbc) {
    return new IdempotencyService(jdbc, Duration.ofHours(24));
}

// En controller:
@PostMapping("/depositos")
public ResponseEntity<?> registrarDeposito(
    @RequestHeader("X-Idempotency-Key") String idempotencyKey,
    @RequestBody DepositoRequest request) {
    // El servicio busca si ya procesó esta key
}
```

## VIOLACIONES MEDIAS (🟡)

### 10. Fórmula de Rendimiento No Capitaliza Intereses

- **[CRITICIDAD]:** 🟡 MEDIA
- **[CATEGORÍA]:** Arquitectura - Business Logic
- **[DESCRIPCIÓN]:** Las especificaciones dicen:
> "Capitalización: mensual" y la fórmula es `Rendimiento = Capital × (Tasa Anual / 365) × Días Transcurridos`

Pero no se define qué sucede con el rendimiento generado. ¿Se suma al capital para el siguiente período? Si no se capitaliza, el rendimiento sería lineal y no exponencial como dicta la capitalización mensual.

- **[IMPACTO]:** Los rendimientos calculados serían incorrectos, afectando el saldo de los clientes.

- **[RECOMENDACIÓN]:** Definir claramente:
```java
// En RendimientoService.calcularRendimientoMensual()
BigDecimal capitalConRendimiento = capitalBase.add(rendimientoAcumulado);
BigDecimal nuevoRendimiento = capitalConRendimiento
    .multiply(tasaInteresMensual)
    .multiply(BigDecimal.valueOf(diasTranscurridos));
```

### 11.密码哈希 Inconsistente Entre Módulos

- **[CRITICIDAD]:** 🟡 MEDIA
- **[CATEGORÍA]:** Arquitectura - Consistencia
- **[DESCRIPCIÓN]:** `SecurityConfig` usa BCrypt (strength 12), pero `Argon2Hasher` usa PBKDF2. La especificación no define cuál usar como estándar. Además, `Argon2Hasher` tiene fallback a SHA-256 puro que debilita toda la seguridad.

- **[IMPACTO]:** Si se migra usuarios de BCrypt a Argon2Hasher, el fallback SHA-256 activaría y expondría contraseñas.

- **[RECOMENDACIÓN]:** Eliminar fallback SHA-256 y unificar a Argon2id (no PBKDF2). Si falla, debe ser excepción no fallback.

## ARQUITECTURA - OBSERVACIONES GENERALES

### ✅ Lo Que Está Bien:
1. **Paquete correcto:** `com.tufondo.ahorros` sigue convención del proyecto
2. **DDD Structure:** Entidades bien definidas con Value Objects
3. **UUID como ID:** Correcto para分散 sistemas
4. **NUMERIC(15,4):** Precisión correcta para dinero
5. **Clean Architecture:** Separación de capas visible en estructura de paquetes

### ⚠️ Concern Arquitectónico:
1. **Acoplamiento con módulo Socios:** Las especificaciones declaran "Dependencias: Módulo socios" pero no se define cómo validar que un socio existe antes de crear cuenta - ¿repository compartido o eventos?
2. **Transaccionalidad distribuida:** Si un depósito falla después de actualizar saldo, ¿cómo se maneja el rollback? ¿Sagas?
3. **Event Sourcing:** Los movimientos son audit trail natural, ¿se considera event sourcing para Rendimientos?
