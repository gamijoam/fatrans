## VIOLACIONES CRÍTICAS (🔴)

### 1. CREDENCIALES CON FALLBACKS INSEGUROS EN DOCKER-COMPOSE

- **[CRITICIDAD]**: 🔴 CRÍTICA  
- **[CATEGORÍA]**: Seguridad - OWASP A02:2021 - Cryptographic Failures  
- **[ARCHIVO]**: `infrastructure/docker-compose.yml:27-28, 50-51`  
- **[DESCRIPCIÓN]**: Las variables MINIO_ACCESS_KEY y MINIO_SECRET_KEY tienen valores por defecto fallback (`minioadmin` / `minioadmin123`). Aunque la configuración usa `${VAR:?error}` para forzar valores requeridos en runtime, los fallbacks debilitan la seguridad si alguien despliega sin establecer estas variables conscientemente.

```yaml
# LÍNEAS 27-28
MINIO_ROOT_USER: ${MINIO_ACCESS_KEY:-minioadmin}
MINIO_ROOT_PASSWORD: ${MINIO_SECRET_KEY:-minioadmin123}
```

- **[IMPACTO]**: Si un desarrollador olvida configurar las variables de entorno, MinIO se despliega con credenciales públicas conocidas. En un entorno de producción, esto permite acceso no autorizado al bucket de documentos KYC.

- **[CORRECCIÓN]**:
```yaml
# Cambiar a valores que fallen si no están configurados
MINIO_ROOT_USER: ${MINIO_ACCESS_KEY:?MINIO_ACCESS_KEY es requerido}
MINIO_ROOT_PASSWORD: ${MINIO_SECRET_KEY:?MINIO_SECRET_KEY es requerido}
# Remover valores por defecto de application.yml líneas 50-51
```

### 2. SECRETOS DE DESARROLLO EN ARCHIVO .ENV COMPROMETIDO

- **[CRITICIDAD]**: 🔴 CRÍTICA  
- **[CATEGORÍA]**: Seguridad - OWASP A02:2021 - Cryptographic Failures  
- **[ARCHIVO]**: `infrastructure/.env:11`  
- **[DESCRIPCIÓN]**: El archivo `.env` contiene `JWT_SECRET=Desarrollo2026SecretoBackendMinimo256BitsParaDesarrolloLocal`. Este archivo NO está en `.gitignore` (solo `.env` general está ignorado pero el de infrastructure tiene su propio archivo).

```bash
# infrastructure/.env
JWT_SECRET=Desarrollo2026SecretoBackendMinimo256BitsParaDesarrolloLocal
```

- **[IMPACTO]**: Si el archivo `.env` de infrastructure se filtra o hace commit accidental, el JWT_SECRET de desarrollo podría ser utilizado para generar tokens válidos. Aunque es un valor de desarrollo, demuestra una práctica insegura de manejo de secretos.

- **[CORRECCIÓN]**:
1. Crear `infrastructure/.gitignore` con contenido:
```
.env
.env.local
.env.*.local
```

2. En `application.yml` línea 33, forzar que JWT_SECRET sea requerido:
```yaml
jwt:
  secret: ${JWT_SECRET:?JWT_SECRET es requerido}
```

3. Nunca almacenar secretos reales en archivos `.env` versionados.

## VIOLACIONES ALTAS (🟠)

### 3. debugPrint EXPONE TOKENS Y DATOS DE USUARIO

- **[CRITICIDAD]**: 🟠 ALTA  
- **[CATEGORÍA]**: Seguridad - OWASP A09:2021 - Security Logging Failures  
- **[ARCHIVO]**: `frontend-mobile/lib/core/auth/repositories/auth_repository_impl.dart:39,43,46` y múltiples archivos  
- **[DESCRIPCIÓN]**: Se encontró uso de `debugPrint` en múltiples ubicaciones que expone información sensible:

```dart
// auth_repository_impl.dart
debugPrint('Login successful for user: ${user.nombreUsuario}');
debugPrint('Login web successful for user: ${user.nombreUsuario}');
debugPrint('Token refresh successful');
```

- **[IMPACTO]**: En dispositivos de desarrollo o si se filtra un log de producción, un atacante podría obtener nombres de usuario válidos. En combinación con otros datos, podría facilitar ataques de credential stuffing.

- **[CORRECCIÓN]**: Utilizar un logger conditional que solo imprima en modo debug:
```dart
// Crear utility/log.dart
class AppLogger {
  static void debug(String message) {
    if (kDebugMode) {
      debugPrint(message);
    }
  }
}

// En producción, todos los debugPrint deben:
// 1. No exponer PII (nombres de usuario, emails)
// 2. No exponer tokens o refresh tokens
// 3. Solo exponer IDs anonimizados para tracking
```

### 4. JWT ALMACENADO EN localStorage - VULNERABLE A XSS

- **[CRITICIDAD]**: 🟠 ALTA  
- **[CATEGORÍA]**: Seguridad - OWASP A01:2021 - Broken Access Control  
- **[ARCHIVO]**: `frontend-web-public/src/lib/auth.ts:14-29` y `frontend-web-public/src/pages/admin/dashboard.astro:77-78`  
- **[DESCRIPCIÓN]**: Los tokens JWT se almacenan en localStorage sin protección HttpOnly cookie:

```typescript
// auth.ts
localStorage.setItem(TOKEN_KEY, result.accessToken);
localStorage.setItem(REFRESH_TOKEN_KEY, result.refreshToken);
```

- **[IMPACTO]**: Si existe cualquier vulnerabilidad XSS en la aplicación, un atacante puede robar los tokens JWT completos. Esto es especialmente crítico porque:
  - `dashboard.astro` hace `JSON.parse(adminUserStr)` sin sanitización (línea 83)
  - No hay protección CSRF visible en las requests

- **[CORRECCIÓN]**: Implementar HttpOnly cookies para tokens:

```typescript
// Usar cookies con httpOnly y secure flags
document.cookie = `admin_token=${token}; HttpOnly; Secure; SameSite=Strict; path=/`;
```

Y en el backend, validar token desde cookie en lugar de header Authorization.

### 5. CORS CON withCredentials PERMITE ORÍGENES LOCALHOST AMPLIOS

- **[CRITICIDAD]**: 🟠 ALTA  
- **[CATEGORÍA]**: Seguridad - OWASP A02:2021 - Broken Access Control  
- **[ARCHIVO]**: `frontend-mobile/lib/core/api/api_client.dart:47-49` y `backend/.../SecurityConfig.java:68-77`  
- **[DESCRIPCIÓN]**: La configuración permite CORS con credenciales desde múltiples localhost:

```java
// SecurityConfig.java línea 68
String envOrigins = System.getenv().getOrDefault("CORS_ORIGINS", 
  "http://localhost:3000,http://localhost:18081,http://localhost:8081");
```

```dart
// api_client.dart línea 47-49
if (kIsWeb) {
  options.extra['withCredentials'] = true;
}
```

- **[IMPACTO]**: Un atacante podría crear un sitio malicioso en localhost:3000 que robe credenciales de un usuario administrativo. Aunque `withCredentials` requiere que el origen esté explícitamente permitido, la lista amplia de localhostsapproved aumenta la superficie de ataque.

- **[CORRECCIÓN]**: Reducir la lista de orígenes permitidos y validar en backend que el origin header coincide exactamente con un dominio esperado (no wildcards).

## MEJORAS RECOMENDADAS (🟡)

### 6. LOGS DE DEBUG EN CÓDIGO DE PRODUCCIÓN

- **[CRITICIDAD]**: 🟡 MEDIA  
- **[CATEGORÍA]**: Mantenibilidad  
- **[ARCHIVO]**: Múltiples archivos en `frontend-mobile/lib/`  
- **[DESCRIPCIÓN]**: Existe aproximadamente 109 instancias de `debugPrint` en el código. Aunque están protegidos por `kDebugMode` en Flutter, en la build web estos podrían comportarse de manera diferente.

- **[IMPACTO]**: Potencial filtración de información sensible en logs de producción si el código se despliega sin las verificaciones adecuadas.

- **[CORRECCIÓN]**: Crear un wrapper de logging centralizado que sea fácilmente desactivable.

---

### 7. FALLBACK CORS PERMITE localhost EN PRODUCCIÓN

- **[CRITICIDAD]**: 🟡 MEDIA  
- **[CATEGORÍA]**: Arquitectura  
- **[ARCHIVO]**: `backend/.../SecurityConfig.java:73-75`  
- **[DESCRIPCIÓN]**: Si `CORS_ORIGINS` está vacío, se usan valores por defecto con localhost:

```java
if (allowedOrigins.isEmpty()) {
    allowedOrigins = List.of("http://localhost:3000", "http://localhost:18081", "http://localhost:8081");
}
```

- **[IMPACTO]**: Un despliegue accidental sin configuración CORS permitiría cualquier sitio en localhost hacer requests autenticadas.

- **[CORRECCIÓN]**: En producción, esta configuración debe fallar si no se configuran orígenes explícitamente. Implementar validación en startup que rechace valores localhost en entornos production.
