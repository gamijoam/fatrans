### 🔴 CRÍTICO-02: Regex de correo permite puntos consecutivos inválidos

**Archivo:** `src/main/java/com/tufondo/socios/application/validation/ValidadorCorreoElectronico.java` (línea 14-16)

```java
private static final Pattern EMAIL_PATTERN = Pattern.compile(
        "^[A-Za-z0-9+_.-]+@[A-Za-z0-9.-]+\\.[A-Za-z]{2,}$"
);
```

**Problema:** El regex `^[A-Za-z0-9+_.-]+` permite puntos consecutivos sin restricción. Correos como `us..er@gmail.com` o `.usuario@email.com` son aceptados como válidos.

**Impacto:** Validación de formato decorreo laxa, permite entradas que rechazarían servidores SMTP reales.

**Corrección sugerida:**
```java
private static final Pattern EMAIL_PATTERN = Pattern.compile(
    "^[A-Za-z0-9+_.-]+(\\.[A-Za-z0-9+_.-]+)*@[A-Za-z0-9.-]+\\.[A-Za-z]{2,}$"
);
```

### 🔴 CRÍTICO-03: Nullable boolean en Socio.activar() puede causar NPE

**Archivo:** `src/main/java/com/tufondo/socios/domain/model/Socio.java` (líneas 57-65)

```java
public void activar() {
    this.eliminado = false;  // ← Asigna false a Boolean que puede ser null
    this.estado = EstadoSocio.ACTIVO;
    this.fechaActivacion = LocalDateTime.now();
    this.fechaDesactivacion = null;
    this.motivoDesactivacion = null;
}
```

**Problema:** Si `this.eliminado` es `null` (nunca inicializado), asignar `false` funciona, pero el método `estaActivo()` usa `Boolean.FALSE.equals(this.eliminado)` que trata null como false (correcto), y `marcarEliminado()` asigna `true`. Sin embargo, hay inconsistencias:

1. `desactivar()` NO modifica `eliminado` (permite inactivo sin eliminar)
2. `marcarEliminado()` solo asigna `eliminado=true` sin cambiar estado
3. Un socio podría quedar en estado `INACTIVO` con `eliminado=true`

**Escenario problemático:**
```
1. socio.desactivar() → estado=INACTIVO, eliminado=null
2. socio.marcarEliminado() → estado=INACTIVO, eliminado=true
3. socio.activar() → estado=ACTIVO, eliminado=false (OK)
```

Pero si alguien llama `buscarPorId()` obtiene un socio INACTIVO+eliminado=true que aún está "visble" en listados.

**Recomendación:** En SocioRepositoryImpl, `buscarPorId()` debería también filtrar por estado ACTIVO si se espera solo activos, o dejar claro en documentación que `buscarPorId()` retorna cualquier socio no-eliminado lógicamente.

### 🟡 ERROR-04: Validación de transición de estado con catch vacío

**Archivo:** `src/main/java/com/tufondo/socios/application/usecase/ActualizarSocioUseCase.java` (líneas 95-101)

```java
private void validarCambioDeEstado(Socio socio, EstadoSocio nuevoEstado) {
    EstadoSocio actual = socio.getEstado();
    boolean transicionValida = switch (actual) {
        case ACTIVO -> nuevoEstado == EstadoSocio.INACTIVO
                || nuevoEstado == EstadoSocio.PENDIENTE_APROBACION;
        case INACTIVO -> nuevoEstado == EstadoSocio.ACTIVO;
        case PENDIENTE_APROBACION -> nuevoEstado == EstadoSocio.ACTIVO
                || nuevoEstado == EstadoSocio.INACTIVO;
    };
    if (!transicionValida) {
        throw new EstadoSocioInvalidoException(socio.getId(), "cambiar estado de "
                + actual + " a " + nuevoEstado);
    }
}
```

**Problema:** Según la documentación de `EstadoSocio.java`, desde `PENDIENTE_APROBACION` solo se puede ir a `ACTIVO` o `INACTIVO`. La implementación actual es correcta. Sin embargo, falta la transición `ACTIVO → PENDIENTE_APROBACION` (reubicar para revisión).

**Observación:** El switch expression está correcto, pero sería más mantenible con un mapa de transiciones validadas en el enum `EstadoSocio` para centralizar la lógica.

### 🟡 ERROR-05: Inconsistencia en validación de correo entre DTOs

**Archivo:** `ActualizarSocioDTO.java` (línea 46) vs `CrearSocioRequestDTO.java` (línea 50)

```java
// En ActualizarSocioDTO.java:
@Email(message = "Formato de correo electrónico inválido")
private String correoElectronico;

// En CrearSocioRequestDTO.java:
// Sin @Email, se delega a ValidadorCorreoElectronico
private String correoElectronico;
```

**Problema:** En `ActualizarSocioUseCase`, si se proporciona correo, se valida DOS veces:
1. Primero por `@Email` de Spring (en el binding)
2. Luego por `ValidadorCorreoElectronico.validarYNormalizar()`

Esto puede causar mensajes de error inconsistentes o confusión.

**Recomendación:** Remover `@Email` de `ActualizarSocioDTO` y delegar toda validación a los validadores de aplicación, igual que en `CrearSocioRequestDTO`.
