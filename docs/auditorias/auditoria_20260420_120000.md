## VIOLACIONES CRÍTICAS (🔴)

### 1. Logout Sin Await - Estado Local Desincronizado

- **[CRITICIDAD]:** 🔴 CRÍTICA
- **[CATEGORÍA]:** Seguridad
- **[DESCRIPCIÓN]:** En `auth_bloc.dart:104-110`, el método `_onLogoutRequested` cambia el estado a `Unauthenticated` ANTES de completar el logout en el backend. La llamada a `_logoutUseCase()` no utiliza `await`.

```dart
// auth_bloc.dart:104-110
Future<void> _onLogoutRequested(
  LogoutRequested event,
  Emitter<AuthState> emit,
) async {
  emit(const Unauthenticated());  // ⚠️ Estado cambiado ANTES del logout
  _logoutUseCase();               // ⚠️ Sin await - fire and forget
}
```

- **[IMPACTO]:** Un atacante con acceso al dispositivo podría mantener una sesión válida en el backend aunque la app muestre "logged out". Los tokens JWT tardan en invalidarse en el servidor, y si el logout falla silenciosamente, el usuario permanece autenticado en el servidor mientras cree estar desconectado. Además, si el logout lanza excepción, el estado local ya fue modificado inconsistentemente.

- **[RECOMENDACIÓN]:** Corregir el flujo para esperar el resultado del logout antes de cambiar el estado:

```dart
Future<void> _onLogoutRequested(
  LogoutRequested event,
  Emitter<AuthState> emit,
) async {
  await _logoutUseCase();  // ✅ await primero
  emit(const Unauthenticated());  // ✅ cambiar estado después
}
```

## VIOLACIONES DE ARQUITECTURA / ALTA (🟠)

### 2. Potential IDOR - No Validación de Propiedad de Cuentas

- **[CRITICIDAD]:** 🟠 ALTA
- **[CATEGORÍA]:** Seguridad - OWASP A01:2021 Broken Access Control
- **[DESCRIPCIÓN]:** El endpoint `/cuentas/socio/$socioId` acepta cualquier `socioId` sin validación explícita de que pertenezca al usuario autenticado. Aunque el frontend pasa `user.id`, un atacante podría manipular la solicitud para acceder a datos de otros socios.

```dart
// api_endpoints.dart:53
static String cuentasPorSocio(String socioId) => '/cuentas/socio/$socioId';
```

- **[IMPACTO]:** Si el backend no valida que el `socioId` pedido corresponda al token JWT del usuario, un atacante podría enumerar cuentas de otros socios. Esto viola el principio de menor privilegio y constituye una vulnerabilidad IDOR.

- **[RECOMENDACIÓN]:** Verificar en el backend ( Spring Boot controller):
  1. El endpoint debe extraer el `socioId` del token JWT, no del parámetro request
  2. O validar que el `socioId` del request coincida con el `socioId` del usuario autenticado
  3. Devolver 403 Forbidden si no coincide

### 3. Silently Swallowing Exceptions - Datos Financieros Incompletos

- **[CRITICIDAD]:** 🟠 ALTA
- **[CATEGORÍA]:** Integridad de Datos
- **[DESCRIPCIÓN]:** En `socio_cuentas_repository_impl.dart:36-45`, las excepciones al obtener saldo o movimientos son capturadas y silenciosamente ignoradas con `continue`:

```dart
for (final cuenta in cuentasResponse.cuentas) {
  try {
    final saldo = await obtenerSaldo(cuenta.numeroCuenta);
    cuentasConSaldo.add(CuentaConSaldo(cuenta: cuenta, saldo: saldo));
    totalAhorrado += saldo.saldoDisponible;

    final movimientos = await obtenerMovimientos(cuenta.numeroCuenta, limit: 3);
    todosMovimientos.addAll(movimientos.movimientos);
  } catch (e) {
    continue;  // ⚠️ Ignora errores silenciosamente
  }
}
```

- **[IMPACTO]:** En un sistema financiero, si una cuenta falla al cargar su saldo, el usuario ve un `totalAhorrado` incorrecto (calculado sin esa cuenta) y no recibe notificación del error. Esto podría causar:
  - Decisiones financieras basadas en datos incompletos
  - Falta de confianza en la aplicación
  - Problemas de auditoría por datos faltantes

- **[RECOMENDACIÓN]:** Implementar manejo de errores apropiada:
  ```dart
  for (final cuenta in cuentasResponse.cuentas) {
    try {
      final saldo = await obtenerSaldo(cuenta.numeroCuenta);
      cuentasConSaldo.add(CuentaConSaldo(cuenta: cuenta, saldo: saldo));
      totalAhorrado += saldo.saldoDisponible;
    } catch (e) {
      // Log error pero continuar - mejor parcial que nada
      debugPrint('Error cargando saldo para cuenta ${cuenta.numeroCuenta}: $e');
    }
    
    try {
      final movimientos = await obtenerMovimientos(cuenta.numeroCuenta, limit: 3);
      todosMovimientos.addAll(movimientos.movimientos);
    } catch (e) {
      debugPrint('Error cargando movimientos para cuenta ${cuenta.numeroCuenta}: $e');
    }
  }
  
  // Validar que tenemos datos mínimos para mostrar
  if (cuentasConSaldo.isEmpty && cuentasResponse.cuentas.isNotEmpty) {
    throw Exception('No se pudo cargar ninguna cuenta');
  }
  ```

### 4. DateTime.parse Sin Validación - Potencial Runtime Exception

- **[CRITICIDAD]:** 🟠 ALTA
- **[CATEGORÍA]:** Estabilidad / Validación de Entrada
- **[DESCRIPCIÓN]:** Los factories de los DTOs usan `DateTime.parse()` directamente sin `try-catch`:

```dart
// cuenta_entity.dart:30-32
fechaApertura: json['fechaApertura'] != null
    ? DateTime.parse(json['fechaApertura'] as String)  // ⚠️ Puede lanzar FormatException
    : DateTime.now(),

// cuenta_entity.dart:117-119
fechaConsulta: json['fechaConsulta'] != null
    ? DateTime.parse(json['fechaConsulta'] as String)  // ⚠️ Puede lanzar FormatException
    : DateTime.now(),
```

- **[IMPACTO]:** Si el backend devuelve una fecha en formato inesperado o corrupto, la app crashea al procesar estos datos. En un contexto financiero donde estas respuestas actualizan saldos, esto podría resultar en experiencias de usuario rotas.

- **[RECOMENDACIÓN]:** Usar `DateTime.tryParse()` en su lugar:
```dart
fechaApertura: json['fechaApertura'] != null
    ? DateTime.tryParse(json['fechaApertura'] as String) ?? DateTime.now()
    : DateTime.now(),
```
