## 2. VIOLACIONES CRÍTICAS

### 🔴 VIOLACIÓN #1: RuntimeException Genérica en Repository
**Archivo:** `backend/src/main/java/com/tufondo/auth/infrastructure/persistence/adapter/UsuarioRepositoryImpl.java`  
**Línea:** 55-57

```java
@Override
@Transactional
public void actualizar(Usuario usuario) {
    UsuarioEntity entity = jpaRepository.findById(usuario.id())
        .orElseThrow(() -> new RuntimeException("Usuario no encontrado"));  // ❌ CRÍTICO
```

**Problema:** Se lanza `RuntimeException` genérica en lugar de la excepción de dominio `UsuarioNoEncontradoException` ya definida.

**Impacto:** Viola el principio de dominio rico, rompe trazabilidad de errores y dificulta el manejo específico en capas superiores.

**Corrección requerida:**
```java
.orElseThrow(() -> new UsuarioNoEncontradoException(
    "No se encontró usuario con ID: " + usuario.id()));
```

---

### 🔴 VIOLACIÓN #2: Falta Configuración de Caché en @Modifying Queries
**Archivo:** `backend/src/main/java/com/tufondo/auth/infrastructure/persistence/jpa/UsuarioJpaRepository.java`  
**Líneas:** 26-27, 30-31

```java
@Modifying
@Query("UPDATE UsuarioEntity u SET u.intentosFallidos = :intentos, u.fechaBloqueo = :fechaBloqueo, u.ultimaModificacion = :ahora WHERE u.id = :id")
void actualizarIntentosFallidos(...);

@Modifying
@Query("UPDATE UsuarioEntity u SET u.intentosFallidos = 0, u.ultimaModificacion = :ahora WHERE u.id = :id")
void resetearIntentosFallidos(...);
```

**Problema:** Las queries `@Modifying` sin `clearAutomatically = true` no invalidan la caché de primer nivel de Hibernate. Esto puede causar inconsistencias si se realizan operaciones de lectura inmediatamente después de una actualización.

**Impacto:** Alto - Puede generar estados inconsistentes en la sesión de Hibernate, especialmente en escenarios de alta concurrencia.

**Corrección requerida:**
```java
@Modifying(clearAutomatically = true, flushAutomatically = true)
@Query("UPDATE UsuarioEntity u SET u.intentosFallidos = :intentos...")
void actualizarIntentosFallidos(...);

@Modifying(clearAutomatically = true, flushAutomatically = true)
@Query("UPDATE UsuarioEntity u SET u.intentosFallidos = 0...")
void resetearIntentosFallidos(...);
```

### 🔴 VIOLACIÓN #3: Mismo Problema en SesionJpaRepository
**Archivo:** `backend/src/main/java/com/tufondo/auth/infrastructure/persistence/jpa/SesionJpaRepository.java`  
**Líneas:** 28, 32, 36, 40-41, 47

```java
@Modifying
@Query("UPDATE SesionEntity s SET s.activa = false WHERE s.tokenId = :tokenId")
void invalidarPorTokenId(...);

@Modifying
@Query("UPDATE SesionEntity s SET s.activa = false WHERE s.refreshToken = :refreshToken")
void invalidarPorRefreshToken(...);

@Modifying
@Query("UPDATE SesionEntity s SET s.activa = false WHERE s.usuarioId = :usuarioId")
void invalidarTodasPorUsuario(...);

@Modifying
@Query("UPDATE SesionEntity s SET s.activa = false WHERE s.usuarioId = :usuarioId AND s.tipo = :tipo AND s.ipAddress = :ipAddress AND s.fechaCreacion < :fechaLimite")
void invalidarSesionesAnteriores(...);

@Modifying
@Query("DELETE FROM SesionEntity s WHERE s.fechaExpiracion < :ahora")
void limpiarSesionesExpiradas(...);
```

**Problema:** Todas las queries `@Modifying` carecen de `clearAutomatically = true`.

**Corrección requerida:** Agregar `clearAutomatically = true, flushAutomatically = true` a todas las anotaciones `@Modifying`.

### 🔴 VIOLACIÓN #4: Sin Validación en JwtProperties
**Archivo:** `backend/src/main/java/com/tufondo/auth/infrastructure/configuration/JwtProperties.java`  
**Líneas:** 6-10

```java
public record JwtProperties(
    String secret,
    AccessTokenConfig accessToken,
    RefreshTokenConfig refreshToken,
    String issuer
) {
```

**Problema:** El record no tiene validación de nulos ni valores por defecto. Si alguna propiedad falta en `application.yml`, la aplicación fallará en runtime con `NullPointerException` en puntos no controlados.

**Impacto:** Crítico en producción - sin mensajes de error claros, debugging difícil.

**Corrección requerida:**
```java
public record JwtProperties(
    String secret,
    AccessTokenConfig accessToken,
    RefreshTokenConfig refreshToken,
    String issuer
) {
    public JwtProperties {
        Objects.requireNonNull(secret, "jwt.secret es requerido");
        Objects.requireNonNull(accessToken, "jwt.accessToken es requerido");
        Objects.requireNonNull(refreshToken, "jwt.refreshToken es requerido");
        Objects.requireNonNull(issuer, "jwt.issuer es requerido");
    }
    
    public record AccessTokenConfig(int expirationMinutes) {
        public AccessTokenConfig {
            if (expirationMinutes <= 0) 
                throw new IllegalStateException("expirationMinutes debe ser positivo");
        }
    }
    
    public record RefreshTokenConfig(int expirationDays) {
        public RefreshTokenConfig {
            if (expirationDays <= 0) 
                throw new IllegalStateException("expirationDays debe ser positivo");
        }
    }
}
```

## 3. VIOLACIONES DE ARQUITECTURA

### 🟡 VIOLACIÓN #4: Fecha Límite Hardcodeada
**Archivo:** `backend/src/main/java/com/tufondo/auth/infrastructure/persistence/adapter/SesionRepositoryImpl.java`  
**Líneas:** 63-65

```java
@Override
@Transactional
public void invalidarSesionesAnteriores(UUID usuarioId, String ipAddress, TipoToken tipo) {
    Instant fechaLimite = Instant.now().minusSeconds(60);  // ⚠️ HARDCODED
    jpaRepository.invalidarSesionesAnteriores(usuarioId, tipo, ipAddress, fechaLimite);
}
```

**Problema:** El límite de 60 segundos está hardcodeado. Este valor debería ser configurable o recibirse como parámetro para permitir flexibilidad.

**Impacto:** Menor - Funcionalidad operativa pero inflexible.

**Recomendación:** Mover a configuración:
```yaml
# application.yml
jwt:
  sesion:
    ventana-inicializacion-segundos: 60
```
