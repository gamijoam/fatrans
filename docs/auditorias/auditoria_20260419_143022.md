### 1. ENVÍO DE PASSWORD EN TEXTO PLANO

- **[CRITICIDAD]**: CRÍTICA
- **[CATEGORÍA]**: Seguridad - A02: Cryptographic Failures
- **[ARCHIVO]**: `frontend-mobile/lib/features/auth/domain/entities/solicitud_registro_dto.dart:151`

**Descripción:**
```dart
'password': password,
```
El password se envía directamente en el cuerpo del JSON sin ningún tipo de hash ni encriptación.

**Impacto:**
- **Si la conexión no es HTTPS (o hay TLS bypass)**: La contraseña es interceptable en texto plano en la red.
- **Si hay logs del lado del servidor**: El password queda almacenado en logs en texto plano.
- **Violación de OWASP A02**: En 2021, "Cryptographic Failures" subió al posición #2, precisamente por casos como este.
- **Incumplimiento PCI-DSS**: En un contexto FinTech/bancario, las contraseñas nunca deben viajar en texto plano.
- **Risk de auditoría regulatoria**: Puede atraer sanciones de organismos financieros.

**Corrección requerida:**
El password debe ser transformado (hashed con salt) **antes** de ser enviado al backend, o alternativamente, el backend debe usar un canal de entrega seguro dedicado para credenciales. Lo recomendado es:

```dart
// Opción 1: Hash en el cliente (si el backend lo espera hasheado)
import 'package:crypto/crypto.dart';
import 'dart:convert';

String hashPassword(String password) {
  final bytes = utf8.encode(password + 'SALT_UNICO_POR_APP');
  final digest = sha256.convert(bytes);
  return digest.toString();
}

// En toJson():
'password': hashPassword(password),

// Opción 2: Usar un canal HTTPS dedicado con certificate pinning
// (Esta es la responsabilidad del infrastructure team)
```

**NOTA IMPORTANTE**: Esta es la **falla más crítica** del branch. Debe corregirse antes de merge.

### 2. PASSWORD ALMACENADO EN EQUATABLE PROPS

- **[CRITICIDAD]**: CRÍTICA
- **[CATEGORÍA]**: Seguridad - Manejo de PII
- **[ARCHIVO]**: `frontend-mobile/lib/features/auth/domain/entities/solicitud_registro_dto.dart:118-142`

**Descripción:**
```dart
@override
List<Object?> get props => [
  // ... otros campos ...
  password,  // <-- El password está en props
];
```

**Impacto:**
- El password queda en memoria como parte del objeto de estado (Equatable).
- Si Flutter/BLoC hace comparisons de estado para detectar cambios, el password puede ser parte de esas comparaciones.
- En logs de debug o errores, el objeto puede ser serializado incluyendo el password.
- Equatable usa `==` y `hashCode` basados en `props`, lo que puede filtrar el password en Collections.

**Corrección requerida:**
```dart
@override
List<Object?> get props => [
  primerNombre,
  segundoNombre,
  primerApellido,
  segundoApellido,
  tipoDocumento,
  numeroDocumento,
  fechaNacimiento,
  genero,
  estadoCivil,
  nacionalidad,
  direccion,
  correoElectronico,
  telefonoPrincipal,
  telefonoSecundario,
  contactoEmergencia,
  empresa,
  departamento,
  cargo,
  tipoContrato,
  salario,
  banco,
  numeroCuenta,
  // password NO DEBE estar en props
];
```

**Recomendación adicional**: Crear un DTO separado `SolicitudRegistroConPassword` que contenga el password para enviar al endpoint, y un `SolicitudRegistroDTO` sin el password para persistencia local.

### 3. REGEX DE EMAIL VULNERABLE A CARACTERES INTERNACIONALES

- **[CRITICIDAD]**: ALTA
- **[CATEGORÍA]**: Seguridad - A03: Injection
- **[ARCHIVO]**: `frontend-mobile/lib/shared/widgets/fatrans_widgets.dart:161`

**Descripción:**
```dart
if (!RegExp(r'^[\w-\.]+@([\w-]+\.)+[\w-]{2,4}$').hasMatch(value)) {
  return 'Correo electrónico inválido';
}
```

**Impacto:**
- El regex no soporta emails IDN (Internationalized Domain Names) como `用户@example.com`.
- Caracteres Unicode en el local part pueden evadir la validación y potencialmente causar:
  - Header injection si el email se usa en encabezados SMTP.
  - Rendering issues si se muestra en contextos HTML sin escape.
- No es una vulnerabilidad directa de injection XSS en Flutter, pero viola principio de defense-in-depth.

**Corrección requerida:**
```dart
// Opción 1: Regex más permisivo pero seguro
if (!RegExp(r'^[\w\.\-]+@[a-zA-Z0-9\.\-]+\.[a-zA-Z]{2,}$').hasMatch(value)) {
  return 'Correo electrónico inválido';
}

// Opción 2: Validación más estricta (recomendada para FinTech)
// Validar estructura básica, dejar validación de formato real al backend
if (value == null || value.isEmpty || !value.contains('@')) {
  return 'El correo es requerido';
}
```

### 4. CÓDIGO DUPLICADO: _buildPasswordField vs _buildTextField

- **[CRITICIDAD]**: MEDIA (Code Smell - DRY Violation)
- **[CATEGORÍA]**: Mantenibilidad
- **[ARCHIVO]**: `frontend-mobile/lib/features/auth/presentation/pages/register_page.dart:603-726`

**Descripción:**
El método `_buildPasswordField()` (líneas 656-726) comparte ~70 líneas de código casi idéntico con `_buildTextField()` (líneas 603-654). La única diferencia es:
1. El manejo de `obscureText` según el tipo de controller.
2. El `suffixIcon` con el toggle de visibilidad.

**Impacto:**
- Violación del principio DRY (Don't Repeat Yourself).
- Mantenimiento duplicado: cualquier cambio en el diseño de inputs requiere cambios en ambos métodos.
- Mayor superficie de código para revisar y testear.

**Corrección recomendada:**
```dart
Widget _buildPasswordField({
  required TextEditingController controller,
  required String labelText,
  String? Function(String?)? validator,
}) {
  // Reusar _buildTextField con obscureText dinámico
  return _buildTextField(
    controller: controller,
    labelText: labelText,
    obscureText: controller == _passwordController
        ? _obscurePassword
        : _obscureConfirmPassword,
    validator: validator,
    suffixIcon: IconButton(
      icon: Icon(
        controller == _passwordController
            ? (_obscurePassword ? Icons.visibility_off : Icons.visibility)
            : (_obscureConfirmPassword
                ? Icons.visibility_off
                : Icons.visibility),
      ),
      onPressed: () {
        setState(() {
          if (controller == _passwordController) {
            _obscurePassword = !_obscurePassword;
          } else {
            _obscureConfirmPassword = !_obscureConfirmPassword;
          }
        });
      },
    ),
  );
}
```
