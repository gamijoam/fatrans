## 2. VIOLACIONES CRÍTICAS

### 🔴 CRÍTICO #1: Campos de SesionEntity NO se mapean correctamente al dominio

**Archivo:** `SesionEntity.java` (líneas 87-92)

```java
public com.tufondo.auth.domain.model.Sesion aDominio() {
    return new com.tufondo.auth.domain.model.Sesion(
            this.id,
            this.usuarioId,
            this.refreshToken,
            this.accessTokenExpiracion,
            this.refreshTokenExpiracion
    );
}
```

**Problema:** El método `aDominio()` **ignora completamente** los campos `activo`, `tipoToken`, `fechaCreacion`, y `ultimaActividad` almacenados en la base de datos. Además, el constructor de `Sesion` no recibe estos valores, lo que significa que **se pierde la información de sesión activa/inactiva** en el dominio.

**Impacto:** 
- `Sesion.estaExpirado()` siempre retorna `false` porque `refreshTokenExpiracion` podría no estar correctamente poblado
- La información de sesión activa/inactiva no se propaga al dominio
- **BUG:** Si una sesión está marcada como `activo=false` en DB, al recuperarla vía `aDominio()` retorna una sesión que parece activa pero no lo está

**Corrección requerida:**
```java
public Sesion aDominio() {
    return Sesion.desdeParametros(
        this.id,
        this.usuarioId,
        this.refreshToken,
        this.accessTokenExpiracion,
        this.refreshTokenExpiracion,
        this.activo,
        this.tipoToken,
        this.fechaCreacion,
        this.ultimaActividad
    );
}
```

**Afecta:** `Sesion.java` necesita expandir su constructor para incluir estos campos.

### 🔴 CRÍTICO #2: SesionRepositoryImpl ignora el parámetro `ipAddress`

**Archivo:** `SesionRepositoryImpl.java` (líneas 78-83)

```java
@Override
@Transactional
public void invalidarSesionesAnteriores(UUID usuarioId, String ipAddress, TipoToken tipo) {
    Instant fechaLimite = Instant.now().minusSeconds(ventanaInicializacionSegundos);
    jpaRepository.invalidarSesionesAnteriores(usuarioId, tipo.name(), ipAddress, fechaLimite);
}
```

**Problema:** El parámetro `ipAddress` se pasa al método JPA pero **no se utiliza en la query**. La consulta definida en `SesionJpaRepository.java` línea 39 no filtra por IP:

```java
@Query("UPDATE SesionEntity s SET s.activo = false WHERE s.usuarioId = :usuarioId AND s.tipoToken = :tipoToken AND s.fechaCreacion < :fechaLimite")
void invalidarSesionesAnteriores(...);
```

**Impacto:** 
- Parámetro desperdiciado
- Inconsistencia entre la firma del método (`ipAddress`) y su implementación
- Potential security issue: no se valida que las sesiones a invalidar provengan de la misma IP

**Corrección requerida:** Modificar el Query JPQL para incluir el filtro de IP o eliminar el parámetro de la firma.

### 🔴 CRÍTICO #3: Actualización parcial de UsuarioEntity con pérdida de datos

**Archivo:** `UsuarioRepositoryImpl.java` (líneas 59-74)

```java
@Override
@Transactional
public void actualizar(Usuario usuario) {
    UsuarioEntity entity = jpaRepository.findById(usuario.id())
            .orElseThrow(() -> new UsuarioNoEncontradoException(...));
    
    entity.setNombreUsuario(usuario.nombreUsuario());
    entity.setCorreoElectronico(usuario.correoElectronico());
    entity.setPasswordHash(usuario.passwordHash());
    entity.setNombreCompleto(usuario.nombreCompleto());
    entity.setRol(usuario.rol());
    entity.setSocioId(usuario.socioId());
    entity.setCuentaActiva(usuario.cuentaActiva());
    entity.setUltimaModificacion(Instant.now());
    // ⚠️ FALTAN: intentosFallidos y fechaBloqueo
}
```

**Problema:** La actualización **no incluye los campos `intentosFallidos` y `fechaBloqueo`**. Si un Admin resetea los intentos fallidos de un usuario, y luego se ejecuta `actualizar()`, estos valores se perderán.

**Impacto:** 
- Datos de auditoría de seguridad inconsistentes
- Posible bypass de protección contra brute force

**Corrección requerida:**
```java
entity.setIntentosFallidos(usuario.intentosFallidos());
entity.setFechaBloqueo(usuario.fechaBloqueo());
```

## 3. VIOLACIONES DE ARQUITECTURA

### 🟠 ARQUITECTURA #1: Validator dentro de Repository Adapter

**Archivo:** `SesionRepositoryImpl.java` (línea 18)

```java
@Value("${jwt.sesion.ventana-inicializacion-segundos:60}")
private int ventanaInicializacionSegundos;
```

**Problema:** El adapter de infraestructura **inyecta configuración de JWT** (`@Value`). Esto viola el principio de separación de capas porque:
- El componente de persistencia no debería conocer detalles de configuración de tokens
- Si cambia la estructura de `JwtProperties`, este adapter se rompe
- Mezcla concerns: persistence + security configuration

**Impacto:** Alto acoplamiento. Cualquier cambio en la estructura de configuración JWT impactará esta clase.

**Corrección recomendada:** Crear un `SesionConfig` dedicado en la capa de infraestructura que encapsule estos valores y sea inyectado vía constructor, o crear un servicio de configuración que abstraiga estos detalles.

---

### 🟠 ARQUITECTURA #2: Uso de Enum como Object en Query JPQL

**Archivo:** `UsuarioJpaRepository.java` (líneas 31-35)

```java
@Query("SELECT u FROM UsuarioEntity u WHERE u.rol = :rol AND u.cuentaActiva = true")
List<UsuarioEntity> findByRolAndCuentaActivaTrue(@Param("rol") Object rol);

@Query("SELECT u FROM UsuarioEntity u WHERE u.socioId = :socioId AND u.rol = :rol AND u.cuentaActiva = true")
Optional<UsuarioEntity> findBySocioIdAndRolAndCuentaActivaTrue(@Param("socioId") UUID socioId, @Param("rol") Object rol);
```

**Problema:** Se declara `@Param("rol") Object rol` cuando debería ser `@Param("rol") Rol rol`. La entidad ya tiene `@Enumerated(EnumType.STRING)` por lo que JPA maneja correctamente enums.

**Impacto:** 
- Pérdida de type-safety
- Posible ClassCastException en runtime
- Código menos legible

**Corrección:**
```java
@Query("SELECT u FROM UsuarioEntity u WHERE u.rol = :rol AND u.cuentaActiva = true")
List<UsuarioEntity> findByRolAndCuentaActivaTrue(@Param("rol") Rol rol);
```

## 4. MEJORAS RECOMENDADAS

### 🟡 MEJORA #1: Agregar validación de tamaño máximo a JwtProperties

**Archivo:** `JwtProperties.java`

Los records `AccessTokenConfig` y `RefreshTokenConfig` validan que sean > 0 pero no tienen límite máximo. Un valor de `Integer.MAX_VALUE` podría causar overflow o comportamientos inesperados.

**Recomendación:**
```java
public record AccessTokenConfig(int expirationMinutes) {
    public AccessTokenConfig {
        if (expirationMinutes <= 0) {
            throw new IllegalStateException("jwt.accessToken.expirationMinutes debe ser un número positivo");
        }
        if (expirationMinutes > 1440) { // 24 horas máximo
            throw new IllegalStateException("jwt.accessToken.expirationMinutes no puede exceder 1440 (24h)");
        }
    }
}
```

### 🟡 MEJORA #2: Null-safety en buscarSesionesActivasPorUsuario

**Archivo:** `SesionRepositoryImpl.java` (líneas 53-58)

```java
public List<Sesion> buscarSesionesActivasPorUsuario(UUID usuarioId) {
    List<SesionEntity> entidades = jpaRepository.findByUsuarioIdAndActivoTrue(usuarioId);
    if (entidades == null) {  // ❌ Innecesario
        return List.of();
    }
    return entidades.stream()
            .map(SesionEntity::aDominio)
            .toList();
}
```

**Problema:** `JpaRepository` **nunca retorna `null`** en métodos que devuelven `List`, siempre retorna una lista vacía. Esta validación es innecesaria y puede indicar malentendido del comportamiento de JPA.

**Recomendación:** Eliminar la validación null y dejar:
```java
return jpaRepository.findByUsuarioIdAndActivoTrue(usuarioId)
        .stream()
        .map(SesionEntity::aDominio)
        .toList();
```

### 🟡 MEJORA #3: Inconsistencia en manejo de excepciones en invalidarPorTokenId

**Archivo:** `SesionRepositoryImpl.java` (líneas 68-72)

```java
@Override
@Transactional
public void invalidarPorTokenId(String tokenId) {
    try {
        jpaRepository.invalidarPorId(UUID.fromString(tokenId));
    } catch (IllegalArgumentException e) {
        // Log ignored
    }
}
```

**Problema:** Silently ignora errores. No hay logging (comentario dice "Log ignored" pero no hay código). En producción esto dificulta el debugging.

**Recomendación:** Agregar logging apropiado:
```java
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

private static final Logger log = LoggerFactory.getLogger(SesionRepositoryImpl.class);

@Override
@Transactional
public void invalidarPorTokenId(String tokenId) {
    try {
        jpaRepository.invalidarPorId(UUID.fromString(tokenId));
    } catch (IllegalArgumentException e) {
        log.warn("Token ID inválido formato UUID: {}", tokenId);
    }
}
```

### 🟡 MEJORA #4: DTOs en lugar de exposición directa de modelos en queries

**Archivo:** `SesionJpaRepository.java` y `UsuarioJpaRepository.java`

Las queries `@Query` exponen directamente `SesionEntity` y `UsuarioEntity`. Si en el futuro se necesita cambiar la estructura de estas entidades, las queries podrían fallar sin control en tiempo de compilación.

**Recomendación:** Considerar mover lógica de query compleja a Specifications o usar Projection interfaces si se necesitan campos específicos.

---

### 🟡 MEJORA #5: Falta documentación OpenAPI en exception handlers

**Archivo:** `GlobalExceptionHandler.java`

No hay documentación de los endpoints de error ni ejemplos de respuestas de error estándar. Para una API de autenticación, esto es crítico para los clientes.
