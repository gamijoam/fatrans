## 2. VIOLACIONES CRÍTICAS

### 🔴 CRÍTICO-01: Conflicto de Nombres — Dos Archivos con la Misma Interfaz

**Archivos afectados:**
- `com/tufondo/socios/domain/repository/SocioRepository.java` (líneas 1-68)
- `com/tufondo/socios/domain/repository/propio.java` (líneas 1-72)

**Descripción del problema:**

Ambos archivos declaran la interfaz `SocioRepository` en el mismo paquete `com.tufondo.socios.domain.repository`. Java no permite dos tipos públicos con el mismo nombre fully-qualified en el mismo paquete. **Esto causa un error de compilación.**

**Evidencia:**
```
// SocioRepository.java (línea 16):
public interface SocioRepository { ... }

// propio.java (línea 13):
public interface SocioRepository { ... }
```

**Impacto:** El proyecto no compilará. Spring no podrá determinar cuál interfaz usar.

**Solución requerida:** Eliminar `propio.java` y conservar únicamente `SocioRepository.java`, que ya contiene la definición completa con el record `Paginacion`.

---

### 🔴 CRÍTICO-02: Violación de Arquitectura — Controller Inyecta Repository Directamente

**Archivo afectado:**
- `com/tufondo/socios/presentation/controller/SocioController.java` (línea 44)

```java
@RestController
@RequiredArgsConstructor
public class SocioController {
    // ...
    private final SocioRepository socioRepository;  // ❌ VIOLACIÓN
```

**Descripción del problema:**

El `SocioController` inyecta directamente `SocioRepository`, violando el principio de que la capa de presentación solo debe conocer la capa de aplicación (UseCases). Esto rompe la **Dependency Rule** de Clean Architecture.

```
┌─────────────────────────────────────────────────────────┐
│  PRESENTATION (Controller)                              │
│  └── ✅ Solo conoce → Application (UseCases)            │
│  └── ❌ NO debe conocer → Domain (Repository)           │
└─────────────────────────────────────────────────────────┘
```

**Impacto:**
- Acoplamiento directo entre presentación y datos
- Imposibilidad de cambiar implementación de persistencia sin modificar Controllers
- Duplicación de lógica de búsqueda en el Controller (método `obtenerSocioActivo`)

**Código problemático en Controller (líneas 49-56):**
```java
private final SocioRepository socioRepository; // INYECTADO DIRECTAMENTE

@GetMapping("/{id}/activo")
public ResponseEntity<SocioResponseDTO> obtenerSocioActivo(@PathVariable Long id) {
    // Usa repository directamente en vez de UseCase
    return ResponseEntity.ok(obtenerSocioUseCase.ejecutarActivo(id));
}
```

**Nota:** El método `obtenerSocioActivo` en el Controller no usa el `SocioRepository` directamente (ya existe `ObtenerSocioUseCase.ejecutarActivo`), pero la **inyección de `SocioRepository` es innecesaria** y viola las reglas arquitectónicas.

## 3. VIOLACIONES DE ARQUITECTURA

### 🟡 ARQ-01: DTOs con Anotaciones de Validación en Capa de Aplicación

**Archivos afectados:**
- `com/tufondo/socios/application/dto/CrearSocioRequestDTO.java`
- `com/tufondo/socios/application/dto/ActualizarSocioDTO.java`

**Descripción:**

Los DTOs contienen anotaciones `jakarta.validation.constraints.*` mezcladas con lógica de mapeo. Aunque funcionalmente es aceptable, según las reglas de Clean Architecture, los **DTOs de request** podrían vivir en la capa `presentation.dto` para mantener separación estricta.

**Situación actual:**
```
application/dto/
├── CrearSocioRequestDTO.java    ← Contiene @NotBlank, @Email, @Pattern
├── ActualizarSocioDTO.java      ← Contiene validaciones
└── SocioResponseDTO.java        ← Solo datos, sin validaciones ✅
```

**Evaluación:** Esto es una **cuestión de estilo arquitectónico**. La separación actual es tolerable porque Spring Boot unifica validación en Controllers, pero idealmente las validaciones deberían estar solo en `presentation.layer`.

---

### 🟡 ARQ-02: Mapper en Capa de Aplicación Accede Directamente a Campos de Entidad

**Archivo afectado:**
- `com/tufondo/socios/application/usecase/SocioDTOMapper.java` (líneas 21-54)

**Descripción:**

El `SocioDTOMapper` está en `application/usecase/`, pero mapea desde `Socio` (dominio). Esto es correcto ✅. Sin embargo, el `SocioEntity.toDomain()` (línea 164 en SocioEntity.java) realiza el mapeo en la capa `infrastructure`. Esto es una **pequeña mezcla de responsabilidades**, ya que la conversión Entidad→Dominio debería idealmente estar solo en el Adapter.

**Situación actual (aceptable):**
```
SocioDTOMapper (application) → Socio (domain)
SocioEntity.toDomain() (infrastructure) → Socio (domain)
```

**Mejora sugerida:** Mover la lógica `toDomain()` del Entity a una clase `SocioMapper` en infrastructure que sea inyectada por el Repository.
