## VIOLACIONES CRÍTICAS (🔴)

### 1. [CRÍTICA] Validación de colateral NO implementada en aprobación

- **Archivo:** `application/usecase/AprobarSolicitudUseCase.java:51-63`
- **Descripción:** Al aprobar una solicitud, se calcula el colateral requerido pero NO se verifica que el socio tenga saldo disponible en su cuenta de ahorro para cubrirlo.
- **Impacto:** Se pueden aprobar créditos donde el socio no tiene fondos suficientes para el colateral. Violación de RN-C-03.
- **Corrección:**

```java
// En AprobarSolicitudUseCase.java - Agregar validación de colateral

// Primero obtener la cuenta de colateral del socio
if (solicitud.getColateralCuentaId() != null) {
    CuentaGarantia cuenta = cuentaGarantiaRepository.buscarPorId(solicitud.getColateralCuentaId())
        .orElseThrow(() -> new ColateralInsuficienteException("Cuenta colateral no encontrada"));
    
    BigDecimal saldoDisponible = cuentaGarantiaRepository.obtenerSaldoDisponible(solicitud.getColateralCuentaId());
    
    if (saldoDisponible.compareTo(colateralRequerido) < 0) {
        throw new ColateralInsuficienteException(
            String.format("Saldo insuficiente: %s < Requerido: %s", saldoDisponible, colateralRequerido));
    }
    
    // Retener el colateral
    cuentaGarantiaRepository.retenerSaldo(solicitud.getColateralCuentaId(), colateralRequerido);
}
```

### 3. [CRÍTICA] Validación IDOR no implementada en pagos de cuotas

- **Archivo:** `application/usecase/RegistrarPagoCuotaUseCase.java:39-44`
- **Descripción:** El método recibe `socioIdToken, isAdmin, isCajero, ipOrigen` pero NO los usa para validar que el socio solo pueda pagar sus propias cuotas.
- **Impacto:** Un socio podría pagar cuotas de OTRO socio. Violación directa de seguridad IDOR.
- **Corrección:**

```java
@Transactional
public PagoCuotaResponse ejecutar(UUID cuotaId, PagoCuotaRequest request, 
        Long socioIdToken, boolean isAdmin, boolean isCajero, String ipOrigen) {
    
    // 1. Obtener la amortización para verificar ownership
    Amortizacion amortizacion = amortizacionRepository.buscarPorId(cuotaId)
        .orElseThrow(() -> new CuotaNoEncontradaException(cuotaId));
    
    PlanAmortizacion plan = planRepository.buscarPorId(amortizacion.getPlanId())
        .orElseThrow(() -> new PlanNoEncontradoException(amortizacion.getPlanId()));
    
    SolicitudCredito solicitud = solicitudRepository.buscarPorId(plan.getSolicitudId())
        .orElseThrow(() -> new SolicitudNoEncontradaException(plan.getSolicitudId()));
    
    // 2. Validación IDOR: socio solo puede pagar sus propias cuotas
    if (!isAdmin && !solicitud.getSocioId().equals(socioIdToken)) {
        throw new AccesoNoAutorizadoException("No puede pagar cuotas de otro socio");
    }
    
    // Continuar con el pago...
}
```

### 4. [CRÍTICA] CanalOrigen no tiene validación de Enum

- **Archivo:** `application/dto/PagoCuotaRequest.java:27`
- **Descripción:** El campo `canalOrigen` es `String` con `@NotNull` pero sin validar que sea uno de: `SUCURSAL`, `WEB`, `MOBILE`, `ATM`.
- **Impacto:** Datos inválidos pueden entrar al sistema.
- **Corrección:**

```java
// Opción 1: Crear enum CanalOrigen
public enum CanalOrigen {
    SUCURSAL, WEB, MOBILE, ATM
}

// En DTO:
@NotNull(message = "canalOrigen es requerido")
private CanalOrigen canalOrigen;

// Opción 2: Si String es requerido, agregar validación manual en use case
private static final Set<String> CANALES_VALIDOS = Set.of("SUCURSAL", "WEB", "MOBILE", "ATM");

// En RegistrarPagoCuotaUseCase:
if (!CANALES_VALIDOS.contains(canalOrigen)) {
    throw new IllegalArgumentException("canalOrigen inválido: " + canalOrigen);
}
```

## VIOLACIONES DE ARQUITECTURA (🟠)

### 5. [ALTA] RuntimeException en lugar de excepciones de dominio

- **Archivo:** `application/usecase/RegistrarPagoCuotaUseCase.java:79, 90`
- **Descripción:** Se lanzan `RuntimeException("Plan no encontrado")` y `RuntimeException("Solicitud no encontrada")` en lugar de excepciones de dominio específicas.
- **Impacto:** Inconsistencia en manejo de errores. Las excepciones específicas permiten mejor tracking y response.
- **Corrección:** Crear y usar `PlanNoEncontradoException` y `SolicitudNoEncontradaException`.

---

### 6. [ALTA] Liberación de colateral NO implementada

- **Archivo:** `application/usecase/RegistrarPagoCuotaUseCase.java:91-94`
- **Descripción:** Cuando el crédito se completa (`plan.estaCompletamentePagado()`), el código dice "pendiente de liberar" pero no llama a `cuentaGarantiaRepository.liberarSaldo()`.
- **Impacto:** El colateral queda retenido indefinidamente después de pagar el crédito.
- **Corrección:**

```java
if (plan.estaCompletamentePagado()) {
    plan.marcarFinalizado();
    planRepository.guardar(plan);
    
    // Liberar colateral
    SolicitudCredito solicitud = solicitudRepository.buscarPorId(plan.getSolicitudId())
        .orElseThrow(() -> new SolicitudNoEncontradaException(plan.getSolicitudId()));
    
    if (solicitud.tieneColateral()) {
        cuentaGarantiaRepository.liberarSaldo(
            solicitud.getColateralCuentaId(),
            solicitud.getColateralMontoRetenido()
        );
        log.info("Colateral liberado para solicitud {}", solicitud.getNumeroSolicitud());
    }
    
    eventPublisher.publish(new CreditoFinalizadoEvent(solicitud.getId()));
}
```

### 7. [ALTA] Validación de colateral para score < 70 no implementada

- **Archivo:** `application/usecase/EvaluarSolicitudUseCase.java`
- **Descripción:** La evaluación determina elegibilidad (score >= 50) pero NO valida que si score < 70, se requiera colateral (RN-E-02).
- **Impacto:** Un crédito puede ser aprobado aunque el socio no tenga colateral suficiente cuando su score es bajo.
- **Corrección:** En `EvaluarSolicitudUseCase`, después de calcular score:

```java
// Verificar requerimiento de colateral
if (score < 70 && solicitud.getColateralCuentaId() == null) {
    evaluacion.setMensajeDecision(
        "Elegible pero requiere colateral (score < 70). Agregue cuenta de colateral."
    );
    // No bloquear evaluación, solo informar - la validación real se hace en aprobación
}
```

### 9. [ALTA] Falta validación de cuota <= 30% salario estimado

- **Archivo:** `application/usecase/EvaluarSolicitudUseCase.java`
- **Descripción:** La regla RN-E-03 dice "Cuota mensual <= 30% del salario estimado" pero la evaluación solo usa puntos de capacidad sin validar este límite.
- **Impacto:** Un crédito puede ser aprobado aunque la cuota exceda el 30% del salario del socio.
- **Corrección:** En `EvaluarSolicitudUseCase`:

```java
// Validar capacidad de pago
BigDecimal cuotaEstimada = PlanAmortizacion.calcularCuotaFrances(
    solicitud.getMontoSolicitado(),
    tipoCredito.getTasaInteresAnual(),
    solicitud.getPlazoMeses()
);

BigDecimal limiteCuota = request.getSalarioEstimado().multiply(new BigDecimal("0.30"));

if (cuotaEstimada.compareTo(limiteCuota) > 0) {
    // Reducir puntaje de capacidad o rechazar
    evaluacion.setMensajeDecision(
        "Cuota excede 30% del salario estimado. Cuota: " + cuotaEstimada + ", Límite: " + limiteCuota
    );
}
```

## MEJORAS RECOMENDADAS (🟡)

### 10. [MEDIA] NPE potencial en calcularScore()

- **Archivo:** `domain/model/EvaluacionCrediticia.java:59-65`
- **Descripción:** El método `calcularScore()` tiene guardas null, pero la evaluación en `EvaluarSolicitudUseCase.java:73-75` usa Math.min directamente sin verificar null.
- **Impacto:** Si puntajes son null, el cálculo puede producir valores inesperados.
- **Corrección:** Ya está manejado en el modelo, pero sería bueno agregar validación en el DTO de evaluación.
