### 2. **[CRÍTICA] Score interno manipulable sin auditoría criptográfica**

**Archivo:** Especificación - `EvaluacionCrediticia.scoreInterno` (0-100 pts)

**Descripción:**  
El score interno se calcula con 3 factores:
- Antigüedad: hasta 30 pts
- Historial ahorro: hasta 30 pts  
- Capacidad pago: hasta 40 pts

**PROBLEMA CRÍTICO:** No existe especificación de:
- Algoritmo hash/verificación del cálculo
- Firma digital del resultado
- Registro inmutable de la evaluación
- Monotonicidad (imposibilidad de alterar tras aprobación)

**Impacto:**  
Un administrador malicioso o un atacante con acceso a BD podría:
- Manipular el score para aprobar créditos fraudulentos
- Modificar factores históricos post-aprobación
- Eliminar evidencia de evaluación corrupta

**Corrección:**
```java
// En EvaluacionCrediticia.java - agregar campos de auditoría
@Entity
@Table(name = "evaluaciones_crediticias")
public class EvaluacionCrediticia {
    
    // ... campos existentes ...
    
    @Column(name = "score_hash", length = 64, nullable = false)
    private String scoreHash;  // SHA-256 del cálculo completo
    
    @Column(name = "factores_serializados", columnDefinition = "TEXT", nullable = false)
    private String factoresSerializados;  // JSON con breakdown del score
    
    @Column(name = "firma_verificable", length = 128)
    private String firmaVerificable;  // Firma con clave asimétrica del sistema
    
    @Column(name = "evaluacion_id_original")
    private UUID evaluacionIdOriginal;  // Para detectar modificaciones
    
    @Version  // Optimistic locking
    private Long version;
}
```

> "Al pagar última cuota: liberar saldoRetenido"

### 4. **[CRÍTICA] Posible double-payment en cuotas concurrentes**

**Archivo:** Especificación - Endpoint #12: `POST /api/v1/cuotas/{id}/pago`

**Descripción:**  
El endpoint de pago de cuotas (`CAJERO/ADMIN`) no especifica:
- Mecanismo de locking pesimista u optimista
- Verificación de estado `PENDIENTE` atómicamente
- Idempotencia del pago
- Rollback en caso de fallo

**Impacto:**  
En un escenario de doble-click o pago concurrente:
1. Cajero A procesa pago de cuota #3
2. Cajero B procesa pago de cuota #3 simultáneamente
3. Ambos ven estado `PENDIENTE`
4. Ambos cobran y registran como PAGADA
5. **Double payment** - el socio paga 2x

**Corrección:**
```java
// En Amortizacion.java - usar optimistic locking
@Entity
@Table(name = "amortizaciones")
public class Amortizacion {
    
    @Id
    private UUID id;
    
    @Enumerated(EnumType.STRING)
    @Column(name = "estado", nullable = false)
    private EstadoAmortizacion estado;
    
    @Version  // CRÍTICO: Previene double-payment
    private Long version;
    
    @Column(name = "reference_pago", unique = true)
    private String referenciaPago;  // Idempotency key
}

// En pago use case
@Service
public class RegistrarPagoCuotaUseCase {
    
    @Transactional
    public PagoResponse ejecutar(PagoRequest request) {
        return amortizacionRepository.findByIdWithLock(request.cuotaId())
            .map(amortizacion -> {
                // Doble verificación en memoria
                if (amortizacion.estado() != EstadoAmortizacion.PENDIENTE) {

                // Verificar idempotencia

                    amortizacionRepository.existsByReferenciaPago(request.referenciaPago())) {
                    throw new PagoDuplicadoException();
                }
                
                // Procesar pago...
            })
            .orElseThrow(() -> new CuotaNoEncontradaException());
    }
}
```

### 6. **[ALTA] Enum `EstadoSolicitud` permite transiciones inválidas**

**Archivo:** Especificación - `SolicitudCredito.estado` enum

**Descripción:**  
El enum tiene 6 estados: `PENDIENTE`, `EN_EVALUACION`, `APROBADA`, `RECHAZADA`, `CANCELADA`, `DESEMBOLSADO`

**PROBLEMA:** No se especifican restricciones de transición:
- ¿Puede pasar de `RECHAZADA` a `DESEMBOLSADO`? (NO)
- ¿Puede pasar de `CANCELADA` a `APROBADA`? (NO)
- ¿Puede pasar de `APROBADA` a `EN_EVALUACION`? (NO - debería ser irreversible)

**Impacto:**  
- Un bug o exploit podría aprobar un crédito ya rechazado
- Inconsistencia de estado en la máquina de estados del crédito
- Violación de reglas de negocio

**Corrección:**
```java
// Usar State Pattern con transiciones validadas
public sealed class EstadoSolicitud 
    permits Pendiente, EnEvaluacion, Aprobada, Rechazada, Cancelada, Desembolsado {
    
    public abstract boolean puedeTransicionarA(EstadoSolicitud nuevoEstado);
}

public final class Pendiente extends EstadoSolicitud {
    @Override
    public boolean puedeTransicionarA(EstadoSolicitud nuevoEstado) {
        return nuevoEstado instanceof EnEvaluacion;
    }
}

public final class Aprobada extends EstadoSolicitud {
    @Override
    public boolean puedeTransicionarA(EstadoSolicitud nuevoEstado) {
        // Irreversible - solo puede ir a Desembolsado o Cancelada (con multa)
        return nuevoEstado instanceof Desembolsado;
    }
}

public final class Rechazada extends EstadoSolicitud {
    @Override
    public boolean puedeTransicionarA(EstadoSolicitud nuevoEstado) {
        return false; // Nunca puede transicionar
    }
}
```

### 7. **[ALTA] Número de solicitud predecible `SOL-CRED-YYYY-XXXXXX`**

**Archivo:** Especificación - `SolicitudCredito.numeroSolicitud`

**Descripción:**  
El formato `SOL-CRED-2026-000001` es secuencial y predecible:
- Un atacante puede enumerar todas las solicitudes de un año
- No hay aleatoriedad criptográfica
- Violación de principio de "Security through obscurity"

**Impacto:**  
- Enumeración de solicitudes de otros usuarios
- Posibilidad de acceder a detalles de solicitudes knowing el número
- Información sensible expuesta (montos, estados, datos de garantes)

**Corrección:**
```java
// Usar UUID o número con suficiente entropía
@Entity
@Table(name = "solicitudes_credito")
public class SolicitudCredito {
    
    // Opción 1: UUID como identificador público
    @Id
    private UUID id;  // Internally used
    
    @Column(name = "numero_solicitud", unique = true, nullable = false)
    private String numeroSolicitud;  // Público, pero no secuencial
    
    @PrePersist
    public void generarNumeroSolicitud() {
        // Usar SecureRandom + timestamp
        this.numeroSolicitud = "SOL-CRED-" + 
            LocalDate.now().getYear() + "-" + 
            String.format("%06d", new SecureRandom().nextInt(999999));
    }
}
```

### 9. **[ALTA] Endpoint `/creditos/{numeroSolicitud}` usa identificador público en URL**

**Archivo:** Especificación - Endpoint #13: `GET /api/v1/creditos/{numeroSolicitud}` - "ALL"

**Descripción:**  
El endpoint permite consultar el estado de un crédito usando el número de solicitud como path parameter accesible para "ALL" (cualquier usuario autenticado).

**PROBLEMA:** Un socio podría consultar solicitudes de OTROS socios si conoce el número.

**Impacto:**  
- Fuga de información financiera de otros usuarios
- Violación de LOPDP (datos personales financieros)
- Un socio malicioso puede espiar créditos de competidores

**Corrección:**
```java
// En CreditoController.java
@GetMapping("/api/v1/creditos/{numeroSolicitud}")
public ResponseEntity<CreditoResponse> consultarEstado(
    @PathVariable String numeroSolicitud,
    @Authentication UsuarioActual usuario) {
    
    // Verificar que el usuario tiene acceso a esta solicitud
    return solicitudCreditoRepository
        .findByNumeroSolicitud(numeroSolicitud)
        .map(solicitud -> {
            // Un socio solo puede ver SUS propias solicitudes
            if (usuario.rol() == Rol.SOCIO && 
                !solicitud.socioId().equals(usuario.socioId())) {
                throw new AccesoNoAutorizadoException();
            }
            return creditoMapper.toResponse(solicitud);
        })
        .orElseThrow(() -> new SolicitudNoEncontradaException());
}
```
