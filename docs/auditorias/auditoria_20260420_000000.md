### Veredicto General

La implementación de cookies httpOnly y token rotation tiene **errores semánticos y potenciales** que deben corregirse antes de producción. Si bien el flujo básico funciona, existen debilidades en:

1. **Semántica incorrecta** en el método de rotación de tokens

## VIOLACIONES CRÍTICAS (🔴)

### 1. Token Rotation - Semántica Equivocada

- **[CRÍTICA]**: **Sesion.id() usado como "token ID" en invalidación**
- **[CATEGORÍA]**: Seguridad - Lógica de Negocio
- **[DESCRIPCIÓN]**: En `AuthUseCase.java:122`, el código llama `sesionRepository.invalidarPorTokenId(sesion.id().toString())`. El nombre del método sugiere que invalida "por token ID", pero en realidad está pasando el UUID de la sesión. La entidad `SesionEntity` tiene un campo `id` (UUID) que representa el ID de la sesión, NO un "token ID".

El flujo en `SesionRepositoryImpl.java:91-97` hace:
```java
public void invalidarPorTokenId(String tokenId) {
    jpaRepository.invalidarPorId(UUID.fromString(tokenId));  // Convierte String → UUID
}
```

Y en `SesionJpaRepository.java:26-28`:
```java
@org.springframework.data.jpa.repository.Query("UPDATE SesionEntity s SET s.activo = false WHERE s.id = :id")
void invalidarPorId(UUID id);
```

**El problema:** Se está usando el session ID para invalidar, cuando debería invalidarse por refresh token hash (como sugiere el nombre del método `invalidarPorRefreshToken`). Esto funciona actualmente porque el session ID es un UUID y el método lo trata como tal, pero la semántica es confusa y potencialmente errónea.

- **[IMPACTO]**: Si en el futuro se agregara un campo `tokenId` real a la sesión (para tracking de tokens rotados), esta lógica fallaría silenciosamente. Además, el audit trail no es claro sobre qué se está invalidando.

- **[RECOMENDACIÓN]**: Renombrar el método a `invalidarPorSesionId()` o `invalidarSesion()` para reflejar la realidad, o mejor aún, usar `invalidarPorRefreshToken(refreshTokenHash)` que es más seguro semánticamente ya que invalida específicamente el token que fue usado.

## VIOLACIONES DE ARQUITECTURA (🟠)

### 2. Cookie Refresh Token - Path Potencialmente Problemático

- **[ALTA]**: **Path de cookie `refresh_token` causa confusión**
- **[CATEGORÍA]**: Arquitectura - Diseño de API
- **[DESCRIPCIÓN]**: En `AuthController.java:93` y `:196`, el path de la cookie de refresh token es `/api/v1/auth/refresh`:
```java
.path("/api/v1/auth/refresh")
```

Pero el endpoint que la consume es `/api/v1/auth/refresh-web`. Según RFC 6265, un cookie con path `/api/v1/auth/refresh` debería enviarse a URLs cuyo path comienza con ese prefijo. Esto significa que:
- ✅ Cookie enviada a `/api/v1/auth/refresh-web` (correcto)
- ✅ Cookie enviada a `/api/v1/auth/refresh` (correcto)
- ❌ Cookie enviada a `/api/v1/auth/refresh-something` (potencialmente incorrecto)

- **[IMPACTO]**: El refresh token se envía a cualquier endpoint bajo `/api/v1/auth/refresh*`, lo cual viola el principio de mínimo privilegio. Si en el futuro se agrega un endpoint como `/api/v1/auth/refresh-statistics`, el refresh token se enviaría involuntariamente.

- **[RECOMENDACIÓN]**: Cambiar el path a `/` para que el cookie sea accesible solo donde sea necesario, o usar un prefijo más específico como `/api/v1/auth/`.

### 3. CORS - Posible Bloqueo en Producción

- **[ALTA]**: **CORS configurado desde variable de entorno sin validación**
- **[CATEGORÍA]**: Seguridad - Configuración
- **[DESCRIPCIÓN]**: En `SecurityConfig.java:66`:
```java
String envOrigins = System.getenv().getOrDefault("CORS_ORIGINS",
    "http://localhost:3000,http://localhost:18081,http://localhost:8081");
List<String> allowedOrigins = envOrigins.contains(",")
    ? List.of(envOrigins.split(","))
    : List.of(envOrigins);
```

**Problemas identificados:**
1. No hay validación de que los orígenes sean URLs válidas
2. No hay diferenciación entre entornos (dev vs prod)
3. Los valores por defecto incluyen `localhost` - riesgoso si se desplieg a producción sin configurar

- **[IMPACTO]**: Si `CORS_ORIGINS` no está configurado en producción, la aplicación permitiría requests desde cualquier `localhost`. Un atacante podría levantar un servidor en localhost y realizar ataques CSRF.

- **[RECOMENDACIÓN]**: Implementar validación estricta:
```java
if (allowedOrigins.isEmpty()) {
    throw new IllegalStateException("CORS_ORIGINS debe estar configurado para producción");
}
// Validar formato de URL
for (String origin : allowedOrigins) {
    URI uri = URI.create(origin.trim());
    if (!uri.isAbsolute() || uri.getHost() == null) {
        throw new IllegalArgumentException("Origen CORS inválido: " + origin);
    }
}
```

## MEJORAS RECOMENDADAS (🟡)

### 4. Missing HttpOnly en Cookies - Posible Confusión

- **[MEDIA]**: **No se documenta el flag `secure` para desarrollo local**
- **[CATEGORÍA]**: Mantenibilidad - Documentación
- **[DESCRIPCIÓN]**: En `AuthController.java:82` y `:92`, las cookies se crean con:
```java
.secure(true)
```

Esto significa que en desarrollo local (HTTP localhost), las cookies NO se almacenarán porque el navegador rechaza cookies `secure` en conexiones HTTP no seguras.

- **[IMPACTO]**: Los desarrolladores podrían experimentar fallos de autenticación en local sin entender por qué.

- **[RECOMENDACIÓN]**: Agregar lógica condicional o documentación clara:
```java
// En desarrollo, permitir cookies sin secure flag
boolean isSecure = !"localhost".equals(request.getServerName())
                    && !"127.0.0.1".equals(request.getServerName());
```

## CONCLUSIÓN

La implementación es **parcialmente correcta** pero tiene errores semánticos que podrían causar problemas en escenarios futuros. La corrección más urgente es la semántica del token rotation (uso de session ID en lugar de refresh token hash para invalidación).

**Recomendación:** Corregir los hallazgos 🟠 antes de pasar a producción. El hallazgo 🔴 es crítico para el audit trail y trazabilidad de sesiones.
